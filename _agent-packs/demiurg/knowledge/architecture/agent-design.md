# Шаблон и гайд CLAUDE.md для агента

Полный документ: шаблон файла + правила заполнения + стратегические решения
(скилл/субагент/hook/MCP) + best practices. Demiurg заполняет плейсхолдеры
на основе данных из Brief (после Intake) или из Team Design (TEAM-режим).

---

## Структура файла

```markdown
# {ROLE_NAME}

{IDENTITY — 1-3 предложения. Кто ты, что делаешь, главный принцип.}
{Максимум 300 токенов. Это то что агент видит ВСЕГДА.}

## Контекст

При старте задачи загрузи:
1. `{path/to/context.md}` — текущие приоритеты
2. `{path/to/relevant-index.md}` — навигация по знаниям

{Файлы загружаются ПО ЗАПРОСУ. Не грузи всё подряд — только когда задача требует.}

## Зона ответственности

Ты отвечаешь за:
- {responsibility_1}
- {responsibility_2}
- {responsibility_3}

Ты НЕ отвечаешь за:
- {not_responsible_1} → передай {OTHER_AGENT}
- {not_responsible_2} → передай {OTHER_AGENT}

## Как работать

1. Прочитай context.md — пойми текущие приоритеты
2. {step_2_specific_to_role}
3. После завершения — обнови ops/log.md

## Решения

- **Сам:** {что решает автономно}
- **С обсуждением:** {что требует обсуждения}
- **Эскалация:** {что передаёт наверх}
```

## Правила заполнения

### Identity (первый блок)
- Максимум 300 токенов (примерно 3-5 предложений)
- Формат: "Ты — {роль}. {Что делаешь}. {Главный принцип}."
- НЕ перечисляй все обязанности — только суть
- Пример: "Ты — CTO. Отвечаешь за код, инфраструктуру и деплой всех проектов.
  Принцип: рабочий код важнее красивого кода."

#### Bad / Good

```
BAD: Identity на 800 токенов
  "Ты — CTO. Отвечаешь за код на Python, FastAPI, Docker, также за деплой
   на VPS, настройку nginx, CI/CD, мониторинг, логирование, бэкапы, тестирование..."

GOOD: Identity на 80 токенов
  "Ты — CTO. Код, архитектура, деплой. Стек: Python/FastAPI, Docker.
   Принцип: рабочий код важнее красивого."

BAD: Один CLAUDE.md на 500 строк с inline knowledge
  CLAUDE.md содержит полное описание продукта, ЦА, конкурентов...

GOOD: CLAUDE.md 80 строк + handle-ссылки на knowledge/
  "Загрузи knowledge/brand/positioning.md когда работаешь с текстами"

BAD: Копипаста knowledge в каждый агент
  CTO.md и CMO.md оба содержат описание продукта

GOOD: Один файл knowledge/product.md, оба ссылаются
```

### Контекст (ссылки на файлы)
- Используй Handle-паттерн: указывай ПУТЬ к файлу, не копируй содержимое
- Максимум 3-5 файлов в "обязательном" списке
- Остальное — "по запросу" с описанием когда нужно
- Пример:
  ```
  Обязательно: context.md, projects/<your-app>/CLAUDE.md
  По запросу: knowledge/INDEX.md (когда нужна инфо о продукте)
  ```

### Зона ответственности
- "Отвечаешь за" — 3-5 пунктов, конкретных
- "НЕ отвечаешь" — 2-3 пункта с указанием КОМУ передать
- Это ключевое: чёткие границы = нет пересечений между агентами

### Решения
- 3 уровня автономии: сам / с обсуждением / эскалация
- Конкретные примеры, не абстракции
- Пример: "Сам: выбор библиотеки, рефакторинг. Эскалация: новый сервис, удаление фичи."

## Антипаттерны (чего НЕ делать)

❌ Identity на 1000 токенов — агент потеряется в собственном описании
❌ "Читай все файлы в knowledge/" — контекст закончится на первой задаче
❌ Пересекающиеся зоны без чёткого правила кто главный
❌ "Можешь делать всё" — агент без границ = агент без фокуса
❌ Копировать содержимое файлов внутрь CLAUDE.md вместо ссылок
❌ Описывать процессы других агентов ("CMO должен делать X") — каждый видит только своё

## Примеры для типовых ролей

### Director (корневой)
```
Ты — AI Director. Координируешь команду агентов. Не делаешь работу сам —
роутишь задачи правильному специалисту. Принцип: понял задачу → нашёл агента
→ передал с контекстом.
```

### CTO
```
Ты — CTO. Отвечаешь за код, архитектуру и деплой. Стек: Python/FastAPI,
Node.js, Docker. Принцип: сначала Issue, потом код. Рабочее важнее красивого.
```

### CMO
```
Ты — CMO. Отвечаешь за контент, маркетинг и воронки. Пишешь тексты, создаёшь
стратегии, упаковываешь продукт. Принцип: каждый текст = конкретное действие
читателя.
```

## Когда что использовать (скилл / субагент / hook / MCP)

### Скилл (внутри агента)
- Разные стили/режимы одной работы (например, стили текста)
- Тот же контекст, другая инструкция
- "Один человек мог бы сделать без переключения контекста"
- Пример: Copywriter + скиллы write-post, write-ad, write-email, review

### Субагент (отдельный процесс)
- Независимая проверка (нужна изоляция от основного контекста)
- Другая компетенция (текст + дизайн = два агента)
- Параллельная работа над разными частями
- Пример: написать текст (Copywriter) + проверить факты (Intelligence)

### Hook (автоматика)
- Детерминированные вещи: форматирование, линтинг, git safety
- PreToolUse = гейткипер (блокирует опасные операции)
- PostToolUse = валидатор (проверяет результат)
- Stop = финализатор (обновляет логи)

### MCP / Плагин (внешний инструмент)
- Доступ к внешним сервисам (Notion, Slack, Gmail)
- Базы данных, API
- Файловые операции за пределами workspace

## Progressive Disclosure (3 уровня загрузки)

- **L1: Metadata** (name + description) — ВСЕГДА в контексте (~100 слов)
- **L2: Body** (SKILL.md или CLAUDE.md) — когда триггернулось (до 500 строк)
- **L3: Resources** (references/, knowledge/) — когда body говорит "прочитай X"

CLAUDE.md = 60-150 строк (L2). Больше → вынеси в knowledge/ (L3).

---

## Pushy Descriptions (для скиллов агента)

Description должен быть навязчивым, иначе скилл не триггернётся:
- "Use this skill when..." + 5 конкретных ситуаций
- Синонимы ключевых слов (русские + английские)
- MANDATORY TRIGGERS секция
- DO NOT use when — ловит false-triggers

---

## Structured Handoff (передача задач между агентами)

```
from: кто передаёт
to: кому
task: что делать
brief: тема + угол + формат
decision_context: ПОЧЕМУ именно это (агент не гадает)
output_expected: что ожидаем на выходе
```

Без `decision_context` принимающий агент додумывает контекст и часто промахивается.

---

## Quality Gate (3 уровня проверки)

1. **Self-check** — внутри агента, чеклист перед отдачей (всегда)
2. **Peer review** — другой агент проверяет (для важных)
3. **Approval** — пользователь подтверждает (для критичных: публикация, деплой, траты)

## Best Practices

- **BP-01:** 2 узких агента лучше 1 широкого
- **BP-02:** Context Engineering > Model Selection
- **BP-03:** Explicit State Management (состояние явное, версионированное)
- **BP-04:** Observability First (логи, метрики, traces)
- **Handle-паттерн:** CLAUDE.md содержит ПУТЬ к файлу, не содержимое
- **Одна сессия = одна задача** (/clear между несвязными)
- **Why > MUST** — объясни причину, не приказывай
- **Негация → позитив** — "переиспользуй" вместо "не создавай"
- **2 примера бьют 10 правил**
- **CLAUDE.md растёт только из ошибок** — правило = Claude накосячил дважды

---

## Антипаттерны (расширенный список)

- **AP-01:** Телефонный перевод — передавать задачу без decision_context
- **AP-03:** Дубли ответственности — два агента за одно и то же
- **AP-14:** 6+ агентов без явного Dispatcher = хаос
- **Глава без подчинённых** = бесполезная обёртка
- **CLAUDE.md > 200 строк** = перегружен
- **Inline контент** (содержимое файла скопировано в CLAUDE.md)
