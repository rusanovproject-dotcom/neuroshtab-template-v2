# Как оценить качество скилла — принципы

Хороший скилл экономит токены. Плохой — сжирает контекст и даёт непредсказуемый результат.
Эта фреймворк поможет отличить одно от другого за 30 секунд.

## 30-секундная диагностика

Открой SKILL.md. Посмотри:

| Вопрос | Красный флаг ❌ | Зелёный флаг ✅ |
|--------|------|------|
| Что делает скилл? | Description > 200 слов или туманное | 1-2 предложения, чёткий триггер |
| Сколько строк body? | > 1000 строк, вся инструкция в теле | ≤ 500 строк, тяжёлое в references/ |
| Примеры есть? | Только описание правил, нет примеров | 2-3 конкретных примера Input → Output |
| Как работает выход? | Неясно что должно получиться | Точные критерии приёмки, примеры output |
| References/ есть? | Всё в одном SKILL.md | references/ загружаются по ходу (Progressive Disclosure) |

## 5 осей оценки

### 1. Clarity of Trigger — триггер ясен и специфичен

**Что оценивать:** Будет ли Claude вызывать этот скилл когда нужно? Не будет ли вызывать когда не нужно?

- Недостаток: description неполный, перечисляет только названия (👎 "дашборды") вместо synonyms и use-cases
- Норма: description говорит ЧТО + КОГДА с примерами ситуаций
- Отличка: описаны near-misses (когда НЕ триггерить)

**Метрика:** Сбалансированность. Undertriggering (не срабатывает когда нужно) хуже, чем overtriggering.

### 2. Economy — экономичность контекста

**Что оценивать:** Грузится ли скилл целиком в контекст или работает Progressive Disclosure?

- Недостаток: body > 800 строк, всё в одном файле, дублирование инструкций
- Норма: body 300-600 строк, heavy content в references/
- Отличка: body говорит "если нужно X, читай references/X.md", структура кристальная

**Принцип:** Если Claude не дочитал references/ — это окей, body самодостаточен. Но если references/ нужны всегда — это признак, что скилл слишком широкий.

### 3. Actionability — конкретность шагов

**Что оценивать:** Даёт ли скилл инструкции, которые Claude сразу может выполнить?

- Недостаток: советы в стиле "будь креативен" и "подумай о UX", нет конкретных действий
- Норма: шаги описаны так, что можно выполнить (инструкция может быть вложена в скрипт)
- Отличка: примеры output, точные команды, даже если они parametrized

❌ "Выбери правильные цвета для дизайна"
✅ "Используй цветовую палитру: основной #007AFF, фон #F5F5F5, текст #333333. Контраст проверь через contrast-checker.com"

### 4. Composability — компонуемость

**Что оценивать:** Может ли этот скилл работать с другими скиллами? Читает/пишет ли общее состояние?

- Недостаток: скилл self-contained, результаты в очень специфическом формате, не взаимодействует с другими
- Норма: результаты можно передать в другой скилл (файлы, JSON, markdown)
- Отличка: явно говорит "выход: markdown файл в /tmp/X" (стандартное место), поддерживает pipe-подход

**Паттерн:** Скилл → stdout/файл → следующий скилл.

### 5. Testability — проверяемость

**Что оценивать:** Как узнать что скилл сработал? Есть ли критерии приёмки?

- Недостаток: "результат должен быть хороший", нет примеров, неясные ожидания
- Норма: есть 2-3 примера input → expected output, легко проверить глазом
- Отличка: описаны assertions (программные проверки) + qualitative checks (человеческий review)

**Примеры assertions:**
- "В JSON должны быть поля: title, author, date"
- "Файл создан, размер > 1KB, парсится без ошибок"

## Красные флаги (признаки плохого скилла)

| Флаг | Почему плохо | Как лечить |
|------|------------|-----------|
| Body > 1000 строк | Грузит весь контекст, Claude медленнее | Выделить в references/, оставить summary в body |
| Нет примеров | Claude не понимает что именно ожидается | Добавить 2-3 Input → Output примера |
| Только негации ("не делай X") | Claude активирует концепцию X вместо понимания | Переписать в позитиве: "делай Y вместо X" |
| Description > 300 слов | Слишком broad, недостаточный триггер | Сузить scope, description ≤ 150 слов |
| Весь контент inline | Нет Progressive Disclosure | Выделить heavy content → references/ |
| Нет exit criteria | Неясно когда скилл завершён | Добавить "Скилл завершён когда:" + чеклист |
| Примеры только happy-path | Не готов к исключениям | Добавить пример с ошибкой и recovery |

## Зелёные флаги (признаки хорошего скилла)

| Флаг | Почему хорошо | Где видеть |
|------|------------|-----------|
| Body 200-500 строк | Баланс между полнотой и экономией | SKILL.md ~400 строк |
| Progressive Disclosure | Контекст загружается по надобности | "Для детали X читай references/X.md" |
| Примеры с деталями | Видно exactly что ожидается | "Input: ..., Output: ..." с копируемым текстом |
| Позитивные формулировки | Claude лучше обобщает | "Используй X потому что Y" вместо "Не используй Z" |
| Clear exit criteria | Скилл знает когда остановиться | "Готово когда: 1) файл создан, 2) валиден, 3) примеры работают" |
| scripts/ и templates | Можно переиспользовать | helper.sh, template.md в assets/ |
| Ограниченный scope | Скилл конкретный, не "всё подряд" | "Генерирует markdown таблицы" ≠ "Создаёт любые документы" |

## Принципы поиска хороших скиллов

Когда учишься писать скиллы, вот где брать вдохновение:

1. **Open-source skill库** — если есть, посмотри какие паттерны повторяются
   - Как структурируют большие скиллы? (Progressive Disclosure?)
   - Какие примеры дают в documentation?
2. **Обратный инжиниринг успешных скиллов** — возьми скилл который хорошо работает
   - Почему он работает? Description? Economy? Примеры?
   - Что из его структуры можно применить к новым скиллам?
3. **Near-miss pattern** — найди скилл похожий на твой
   - Чем твой лучше/хуже?
   - Какие решения из похожего скилла подходят твоему?
4. **Человеческий feedback** — когда скилл используют
   - Где Claude перетриггерит его? (Description слишком broad)
   - Где недотриггерит? (Забыл синонимы)
   - Какой контент читает, какой нет? (Economy problem?)

## Рубрика оценки 0-10

Для каждой оси посчитай score:

| Ось | 0-3 Плохо | 4-6 Окей | 7-10 Отлично |
|-----|-----------|----------|------------|
| **Clarity** | Не понять когда триггерить | Основное ясно, edge-cases неясны | Описаны триггеры + near-misses |
| **Economy** | Вся инструкция в body, > 800 строк | 500-700 строк, некоторое дублирование | ≤500 строк, clear separation в references/ |
| **Actionability** | Советы без конкретных шагов | Шаги есть, но vague в деталях | Примеры output, шаги копируемы |
| **Composability** | Результаты уникального формата, нельзя pipe | Результаты в markdown/JSON, но undocumented | Явно сказано где выход, какой формат |
| **Testability** | Нет примеров, ожидания туманны | Примеры есть, но не все edge-cases | Примеры + assertions + human review rubric |

**Финальный score = среднее по 5 осям. ≥7 = годный скилл. ≥8 = школьный пример.**

## Быстрая самопроверка перед публикацией

**Заполни чеклист:**

- [ ] Description ≤ 150 слов, говорит ЧТО и КОГДА
- [ ] Body ≤ 500 строк (или есть references/ и body ≤ 400)
- [ ] Минимум 2 примера Input → Output
- [ ] Все инструкции в положительной форме (нет "не делай")
- [ ] Есть exit criteria: скилл знает когда завершён
- [ ] scripts/ или templates если есть повторяемый код
- [ ] Протестировано на 2+ real user prompts (не на тестовых)

Если ≥ 6/7 — скилл готов.
