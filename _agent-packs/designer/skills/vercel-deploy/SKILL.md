# Vercel Deploy — превращаем готовый HTML в живой сайт

## Зачем

Пользователь получил готовый лендинг / сайт / интерактив как HTML-файл. Дальше два варианта:
1. Файл лежит у него на компьютере, никому не покажешь без отправки zip-архива
2. **Сайт живёт по настоящей ссылке, открываешь с телефона, присылаешь пользователю, индексируется**

Этот скилл делает (2) — за 5 минут, одной командой, бесплатно (Vercel Hobby tier).

---

## Когда запускать

**Автоматически:**
- После создания любого самостоятельного HTML-файла (лендинг, one-pager, сайт-визитка, дашборд)
- **Всегда после секции «После результата»** главного flow Designer'а — предлагай живой URL как следующий логичный шаг

**По прямому запросу:**
- «Задеплой на vercel»
- «Выложи сайт»
- «Опубликуй лендинг»
- «Сделай живой URL»

**Не запускать:**
- Пользователь сказал «мне только файл» / «на свою Tilda буду класть» / «уже есть домен на другой платформе»
- Это Claude Code артефакт для внутреннего демо, не для внешнего мира

---

## Правило подачи пользователю

**Всегда предлагай живой URL живым языком, не техническим.** Не говори «Vercel deployment», «production environment». Говори:
> *«Могу за 5 минут сделать из этого живой сайт по настоящей ссылке — можно открыть с телефона, прислать пользователям, показать в Telegram. Публикация бесплатная. Хочешь?»*

Если пользователь соглашается — выполняй алгоритм ниже.

---

## Алгоритм

### Шаг 1. Проверь готовность среды

Запусти три команды через Bash:

```bash
vercel --version 2>&1 || echo "NOT_INSTALLED"
vercel whoami 2>&1 || echo "NOT_LOGGED_IN"
node --version
```

**Варианты результата:**

**(A) Всё готово** — Vercel CLI установлен, пользователь залогинен.
→ Сразу переходи к Шагу 4.

**(B) Vercel CLI не установлен** (`NOT_INSTALLED` или команда не найдена)
→ Шаг 2A.

**(C) CLI установлен, но пользователь не залогинен** (`NOT_LOGGED_IN`)
→ Шаг 2B.

**(D) Node.js нет** (`node --version` даёт ошибку)
→ Шаг 2C (самый фундаментальный fallback).

---

### Шаг 2A. Установка Vercel CLI

Скажи пользователю:
> *«У тебя пока нет инструмента который выкладывает сайты на Vercel. Установлю за 30 секунд — безопасно, бесплатно. Запущу команду.»*

Запусти:
```bash
npm install -g vercel
```

Если `npm` не найден → переходи к Шагу 2C.
Если установка прошла → попроверь: `vercel --version`.

Дальше переходи к Шагу 3 (логин).

---

### Шаг 2B. Пользователь не залогинен

Скажи пользователю живым языком:
> *«Тебе нужно войти в Vercel — один раз, займёт минуту. Это бесплатный сервис, как Google-аккаунт для сайтов. Сейчас я открою окно входа.»*

Запусти:
```bash
vercel login
```

Эта команда **интерактивная** — она спросит email или предложит войти через GitHub / Google. Скажи пользователю:
> *«Сейчас в терминале появится вопрос — выбери удобный способ входа: GitHub / GitLab / Bitbucket / Email. Google-аккаунт тоже подойдёт через email. Если впервые на Vercel — тебе предложат сразу создать аккаунт в том же окне. Всё бесплатно.»*

После успешного логина проверь:
```bash
vercel whoami
```

Дальше — Шаг 4.

---

### Шаг 2C. Нет Node.js или npm

Скажи пользователю:
> *«Для деплоя нужен Node.js — бесплатная среда от Microsoft/OpenJS. Самый простой путь — скачать и поставить с [nodejs.org](https://nodejs.org). Жми зелёную кнопку «LTS» — установщик сам всё сделает. После этого скажи мне — вернёмся к деплою.»*

Альтернатива — через Homebrew (если пользователь на macOS и знаком с brew):
```bash
brew install node
```

Но по умолчанию **не предполагай** что у пользователя есть brew — предложи простой путь через сайт.

**Останови деплой** — дождись пользователя. Запишу в `memory.md`: «ожидает установку Node.js».

---

### Шаг 3. (встроен в 2B) Логин

После `vercel login` и подтверждения что залогинен — переходим к Шагу 4.

---

### Шаг 4. Подготовка папки

Для деплоя Vercel нужна папка с файлом `index.html`. Если у пользователя один HTML-файл — нужна обёртка.

```bash
# Определи путь к HTML-файлу пользователя
HTML_FILE="/Users/.../artifacts/.../landing.html"

# Создай папку для деплоя
DEPLOY_DIR=$(dirname "$HTML_FILE")/deploy-$(date +%Y%m%d-%H%M)
mkdir -p "$DEPLOY_DIR"

# Скопируй файл как index.html
cp "$HTML_FILE" "$DEPLOY_DIR/index.html"

# Создай минимальный vercel.json (опционально)
cat > "$DEPLOY_DIR/vercel.json" <<'EOF'
{
  "cleanUrls": true,
  "trailingSlash": false
}
EOF

echo "Готово: $DEPLOY_DIR"
```

Скажи пользователю:
> *«Собрал папку для деплоя. Сейчас выкладываю.»*

---

### Шаг 5. Деплой

Запусти из подготовленной папки:
```bash
cd "$DEPLOY_DIR"
vercel deploy --prod --yes --name={project-slug}
```

Где `{project-slug}` — имя проекта. Подбирай из контекста:
- Пользователь работает над основным продуктом «AI-офис» → `ai-office`
- Пользователь делает лендинг кофейни «Корица» → `korica-landing`
- Пользователь собирает one-pager для услуги «AI-коуч» → `ai-coach-offer`

Флаги:
- `--prod` — сразу production (не preview)
- `--yes` — не спрашивать подтверждений по default-настройкам
- `--name` — имя проекта (определит поддомен на `*.vercel.app`)

Дождись окончания. В выводе увидишь:
```
✅ Production: https://{project-slug}-xxxxxxx.vercel.app [copied to clipboard]
```

---

### Шаг 6. Отдай пользователю URL

Живым языком:
> *«Готово. Твой лендинг по живой ссылке:*
>
> *`https://{project-slug}.vercel.app`*
>
> *Уже работает — открой с телефона, убедись. Можешь слать пользователям, постить в Telegram, добавлять в шапку канала.»*

**Если URL длинный / некрасивый** (типа `{project-slug}-abc123.vercel.app`) — скажи:
> *«Ссылка рабочая, но у поддомена есть мусор. Хочешь короче — `{project-slug}.vercel.app` — скажу куда жать в Vercel-дашборде, чтобы переименовать за 30 секунд.»*

---

### Шаг 7. (опционально) Custom domain

Если пользователь говорит «хочу свой домен» (типа `myproject.ru`):
1. Уточни — домен у него уже куплен или нужно купить?
2. Если куплен — отправь на инструкцию:
   > *«Зайди на [vercel.com/dashboard](https://vercel.com/dashboard) → открой проект → Settings → Domains → Add → введи свой домен. Он покажет какие DNS-записи добавить у регистратора — обычно это A-запись или CNAME. Если застрял — скрин мне, помогу.»*
3. Если не куплен — предложи namecheap / reg.ru / cloudflare. Не настаивай — это уже вне зоны дизайна.

---

### Шаг 8. Зафиксируй в памяти

Допиши в `memory.md`:
```
## Decisions
YYYY-MM-DD — Задеплоил лендинг {project-slug} на Vercel
  URL: https://{project-slug}.vercel.app
  Source: /artifacts/brand-test/YYYY-MM-DD_{project-slug}-landing.html
  Deploy dir: /artifacts/brand-test/deploy-YYYYMMDD-HHMM
  Client: [имя пользователя]
```

Это нужно чтобы при следующем апдейте лендинга ты знал куда пушить update.

---

## Апдейт существующего деплоя

Если пользователь говорит «поменяй на лендинге X, обнови живой сайт»:
1. Найди в `memory.md` запись про проект → путь к `deploy-YYYYMMDD-HHMM/`
2. Обнови `index.html` в этой папке (замени файлом новой версии)
3. `vercel deploy --prod --yes` из той же папки
4. Vercel обновит существующий URL за 10-20 секунд, старая ссылка остаётся рабочей
5. Сообщи пользователю: *«Обновил по той же ссылке. Через 10 секунд обнови страницу — увидишь новую версию.»*

---

## Частые ошибки и fallback

| Ошибка | Что делать |
|--------|-----------|
| `Error: No existing credentials found` | → запусти `vercel login` снова |
| `Error: EACCES permission denied` при `npm install -g` | → `sudo npm install -g vercel` (macOS/Linux) или `npm install -g --force vercel` |
| `Project not found` при `--name` | → убери `--name`, Vercel создаст новый проект автоматически |
| Таймаут деплоя | → проверь интернет, попробуй ещё раз. Vercel обычно быстрый |
| URL работает но стили битые | → проверь что `index.html` ссылается на CSS/JS относительными путями, не абсолютными |

## Альтернативы Vercel (если не подошёл)

Если пользователь принципиально не хочет Vercel (например, работает в РФ с VPN-опасениями):
- **SourceCraft Sites** (Yandex, работает в РФ без VPN) — есть у нас skill `sourcecraft-publisher`
- **GitHub Pages** (бесплатно, требует GitHub-аккаунт) — есть skill `github-pages-publisher`
- **Netlify** — аналог Vercel, такая же логика: `netlify deploy --prod`
- **Cloudflare Pages** — бесплатно, быстро, но нужен Cloudflare-аккаунт

Предлагай Vercel по умолчанию — он самый простой + быстрый. Остальные — по запросу пользователя.

## Guardrails

- **Не деплоить без подтверждения пользователя.** Всегда спрашивай «Публикуем на живой URL? Бесплатно, 5 минут.»
- **Не передавай пользовательский HTML в публичный домен без спроса** — он может содержать внутренние ссылки, placeholder-тексты, личные данные.
- **Не используй чужой Vercel-аккаунт.** Всё под аккаунтом пользователя. Если у тебя нет доступа — останови и попроси логин.
- **Проверяй что пользователь видит живой URL и он работает** — не считай деплой успешным пока пользователь не подтвердил.
- **Храни URL в памяти** (`memory.md`) — при следующем апдейте знаешь куда пушить.
- **Защита от случайного прода.** Если деплой — внутренний тест / превью, **не используй `--prod`**. Только когда пользователь сказал «это публичная версия».

## Self-check перед деплоем

- [ ] HTML-файл готов (прошёл через `design-critique`, Design Score ≥ B)
- [ ] Файл не содержит placeholder-текстов («Lorem ipsum», «[вставить текст]», тестовых имён)
- [ ] Пользователь явно сказал «публикуем» / «выкладываем»
- [ ] Vercel CLI установлен и пользователь залогинен (или мы уже прошли Шаги 2A-2B)
- [ ] Имя проекта осмысленное (не `untitled-123`)
- [ ] Папка деплоя собрана (index.html на месте)
- [ ] После деплоя URL открыт и проверен мной / пользователем
- [ ] URL записан в `memory.md` для будущих апдейтов
