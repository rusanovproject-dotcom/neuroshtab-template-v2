# /audit-project — Универсальный аудит AI-офиса

Демиург смотрит на готовый офис (свой или сторонний) и выдаёт диагностический отчёт. Что хорошо, что плохо, что средне — по 5 категориям. Дальше — план улучшений с приоритизацией.

**Принцип:** не громи. Цель — **аккуратно прокачивать**, не ломать. Каждое улучшение должно делать офис **проще для пользователя**, не сложнее.

---

## Когда запускать

| Триггер | Действие |
|---------|----------|
| «Проверь офис», «наведи порядок» | Запускай |
| «Что улучшить в офисе» | Запускай |
| Готовишь новый сборки **на основе** существующего | Запускай (понять что копировать как пример, что — как анти-пример) |
| После большой пересборки (фаза Wire завершена) | Запускай как Quality Gate |
| Минорная правка одного агента | НЕ нужен — слишком тяжёлый процесс |
| Стратегическое решение по офису | НЕ нужен — нужно `/interview` |

---

## Вход

```
Аргумент: путь до офиса
Пример: /audit-project office/
        /audit-project client-office-template/office/
        /audit-project clients/{client}/office/

Опционально:
  --quick      — быстрый аудит (5 мин, только P0)
  --standard   — стандартный (15 мин, P0+P1) — DEFAULT
  --deep       — глубокий (30+ мин, все категории + рекомендации)
```

## Process (5 фаз)

### Фаза 1: Сканирование (5 мин)

```bash
# Базовая инвентаризация
find {office} -name "CLAUDE.md" -not -path "*/archive/*" | head -50
find {office} -name "AGENTS.md" | head -5
find {office} -name "MEMORY.md" | head -50
find {office} -name "context.md" | head -5
find {office} -name "INDEX.md" | head -20

# Существующие validate-скрипты (если есть)
bash demiurg/scripts/validate-office.sh {office} 2>&1 | tee /tmp/audit-validate.log

# Размеры файлов (для проверки правила ≤200 строк)
find {office} -name "CLAUDE.md" -not -path "*/archive/*" -exec wc -l {} \;
```

**Что собрать:**
- Список агентов
- Размеры всех CLAUDE.md
- Структура папок (3 уровня вглубь)
- Какие файлы памяти у каждого агента (CLAUDE.md, MEMORY.md, memory.md, failures.md, soul.md, core.md, overrides.md — какие есть, какие нет)
- Существующие протоколы (`protocols/`)

### Фаза 2: Аудит по 5 категориям

#### Категория 1: Структура (Structure)

Применяется `office-blueprint.md` как эталон.

| Чек | Pass / Warn / Fail |
|-----|-------------------|
| Корневой `CLAUDE.md` существует, ≤200 строк | ✓ / ⚠ (200-300) / ✗ (>300) |
| `AGENTS.md` живая карта (не пустая) | ✓ / ⚠ (пустые поля) / ✗ |
| `context.md` ≤50 строк, обновлён за последние 7 дней | ✓ / ⚠ / ✗ |
| Глубина папок ≤3 уровня | ✓ / ⚠ / ✗ |
| Каждый агент имеет свою папку `agents/{name}/` | ✓ / ⚠ / ✗ |
| `archive/` присутствует и read-only | ✓ / ⚠ / ✗ |
| `ops/log.md` или эквивалент есть | ✓ / ⚠ / ✗ |
| Нет дублей файлов с пересекающимися именами | ✓ / ⚠ / ✗ |

#### Категория 2: Knowledge

Применяется `knowledge-ownership.md` + `context-strategies.md`.

| Чек | Pass / Warn / Fail |
|-----|-------------------|
| `knowledge/INDEX.md` существует, актуален | ✓ / ⚠ / ✗ |
| Каждый knowledge-файл ≤300 строк | ✓ / ⚠ / ✗ |
| Триггеры (когда загружать) указаны для L2-файлов | ✓ / ⚠ / ✗ |
| Нет дублей контента (один факт = одно место) | ✓ / ⚠ / ✗ |
| Граница общий vs agent-specific knowledge явная | ✓ / ⚠ / ✗ |
| Нет inline-копий контента в CLAUDE.md (handle-паттерн) | ✓ / ⚠ / ✗ |
| `evolving/` (или `learnings/`) пополняется | ✓ / ⚠ / ✗ |

#### Категория 3: Routing

Применяется `routing-patterns.md` + `director-cookbook.md`.

| Чек | Pass / Warn / Fail |
|-----|-------------------|
| Director — отдельный агент, не «файл с роутингом» | ✓ / ⚠ / ✗ |
| Director ≤200 строк, не дублирует AGENTS.md | ✓ / ⚠ / ✗ |
| 5 секций Director'а: Roster / Decompose / Delegate / Synthesize / Escalate | ✓ / частично / ✗ |
| Hub-and-spoke, не mesh без обоснования | ✓ / ⚠ (есть peer-to-peer без yaml) / ✗ |
| Точки входа задач документированы (entry points) | ✓ / ⚠ / ✗ |
| Handoff-формат yaml с `objective + expected_output + boundaries` | ✓ / ⚠ (свободный текст) / ✗ |
| Нет пересечений зон (overlap >50%) | ✓ / ⚠ (30-50%) / ✗ (>50%) |

#### Категория 4: Memory

Применяется `memory-architecture.md`.

| Чек | Pass / Warn / Fail |
|-----|-------------------|
| Каждый агент имеет `memory.md` (append-only) | ✓ / ⚠ (часть) / ✗ |
| Каждый агент имеет `failures.md` | ✓ / ⚠ / ✗ |
| `MEMORY.md` (auto-memory) в `.claude/memory/` структурирован по 4 категориям (user/feedback/project/reference) | ✓ / ⚠ / ✗ |
| `MEMORY.md` index ≤200 строк, каждая запись ≤150 символов со ссылкой | ✓ / ⚠ / ✗ |
| Память пополняется: записи за последние 30 дней | ✓ / ⚠ (>30 дней без) / ✗ (вообще пусто) |
| Для client-офиса: per-client memory с `valid_until` | ✓ / N/A / ✗ |
| Нет чувствительных данных в памяти (паспорта, медицина) | ✓ / ⚠ / ✗ |

#### Категория 5: Characters & Identity

Применяется `agent-design.md` (Bad/Good identity, антиклон).

| Чек | Pass / Warn / Fail |
|-----|-------------------|
| Identity каждого агента ≤300 токенов | ✓ / ⚠ / ✗ |
| Каждый агент имеет уникальный принцип (не клон шаблона) | ✓ / ⚠ / ✗ |
| Архетипы агентов уникальны (антиклон) | ✓ / ⚠ (2-3 похожих) / ✗ (массовый клон) |
| Тон/voice описаны конкретно (фразы-маркеры, стоп-слова) | ✓ / ⚠ / ✗ |
| Нет AI-слопа в инструкциях («рад помочь», «отличный вопрос») | ✓ / ⚠ / ✗ |
| Output contract явный (что именно агент возвращает) | ✓ / ⚠ / ✗ |

### Фаза 3: Приоритизация находок

Каждая Fail / Warn получает приоритет:

- **P0 — критичные** (мешают работать прямо сейчас): broken links, агент без AGENTS.md, > 5 пересекающихся зон, MEMORY.md > 500 строк, конфликтующие триггеры скиллов
- **P1 — серьёзные** (мешают расти): отсутствует Director как first-class агент, knowledge без триггеров (Progressive Disclosure нарушена), память не пополняется, нет handoff yaml
- **P2 — мелочи и cleanup**: старые комменты, дубли мелких файлов, опечатки, не-критичные warnings

### Фаза 4: План улучшений

Для каждой находки P0/P1 — **конкретный action**:
- Файл который править / создавать / удалять
- Точный шаг (а не «улучшить структуру»)
- Время на исправление (estimate)
- Риски при правке

Pattern: **аккуратно прокачивать**. Не ломаем то что работает. Всегда — git snapshot перед правками.

## Output contract

Скилл возвращает:
1. **Audit report** в `{office}/audits/YYYY-MM-DD-audit.md`
2. **Score** общий (0-100) + по 5 категориям
3. **План улучшений** с фазами и временем
4. **Сообщение пользователю**: `Score: X/100. Главное: {3 пункта}. Полный отчёт: {путь}`

## Эталоны (читай перед запуском)

- `knowledge/architecture/office-blueprint.md` — что считать за «правильно» (Structure)
- `knowledge/architecture/memory-architecture.md` — Memory эталон
- `knowledge/architecture/routing-patterns.md` — Routing эталон

## Self-check (Demiurg перед отдачей отчёта)

- [ ] Прочитал все 5 архитектурных knowledge-файлов выше
- [ ] Запустил `validate-office.sh` (или эквивалент)
- [ ] Каждая P0 имеет конкретный action с путём и временем
- [ ] Не ломаю то что работает — есть секция «Что НЕ трогать»
- [ ] Score обоснован (не «потолочный»)
- [ ] План улучшений приоритизирован по revenue impact (если применимо)
- [ ] Сравнил с эталоном (`client-office-template/`)
- [ ] Отчёт сохранён в `{office}/audits/`

---

## Quality Gate

Перед отдачей пользователю:
- [ ] Score не «по доброте» — каждый чек обоснован
- [ ] P0 действительно критичны (не «было бы круто»)
- [ ] План работает — не противоречит сам себе
- [ ] Учитывает тип офиса (dev / marketing / client / mixed) — не одно правило для всех

---

## Anti-patterns (как НЕ аудитить)

| Анти-паттерн | Почему плохо |
|--------------|--------------|
| Громить всё что не идеально | пользователь получит 200-пунктовый список и забьёт. Цель — аккуратная прокачка, не катастрофизация |
| Применять dev-критерии к marketing-офису | Например, требовать «test coverage» от Copywriter'а — глупо |
| Не учитывать что офис growing | Молодой офис ≠ зрелый. P2 для зрелого = P1 для молодого |
| Игнорировать «работает но некрасиво» | Если работает — оставь. Прокачка только если есть **реальная** боль |
| Игнорировать соул-слой | Identity / characters — половина качества офиса. Если все агенты звучат одинаково — это P0 |
| Скоринг без цифр | «Хорошо» — не оценка. Нужны 0-100 баллов с обоснованием |
