# /strategist-unpack — Фаза 2 UNPACK

Вторая фаза Стратега. После INTAKE уже есть черновики и 3-5 подтверждённых тезисов. Цель UNPACK — детально извлечь из документов всё ценное, довести черновики до рабочего состояния, классифицировать основную слабую сторону (из 10 типов) и зафиксировать противоречия.

Поведение и тон — из `office/agents/strategist/core.md`.

## Когда запускается
- После `/strategist-intake` пользователь подтвердил базовые тезисы
- Команды "/strategist-unpack", "распакуй глубже", "пройди детально по документам"
- Если `diagnostic-card.md` заполнена только гипотезами и нет классификации слабой стороны

## Пререквизиты
- Существуют черновики от INTAKE в `projects/{{PROJECT_NAME}}/`. Если их нет — сначала запустить `/strategist-intake` (предложить пользователю).
- Доступны `inbox/prework.md` и `inbox/docs/*`.

## Шаги

### 1. Перечитать с полным контекстом
Перечитать все документы из `inbox/docs/` и `inbox/prework.md` ещё раз, уже держа в голове подтверждённые пользователем тезисы из INTAKE. Теперь ищутся не первые впечатления, а детали: цифры, фразы ЦА, механики оффера, конкретные кейсы.

### 2. Детально извлечь по типам документов

Для каждого типа документа — свой набор артефактов.

**Лендинг / посадочная →**
- `projects/{{PROJECT_NAME}}/audience/segments.md` — конкретные сегменты (для кого страница)
- `projects/{{PROJECT_NAME}}/audience/pains.md` — минимум 5 болей на сегмент, дословно как написано
- `projects/{{PROJECT_NAME}}/audience/jobs-to-be-done.md` — JTBD, если просматриваются
- `projects/{{PROJECT_NAME}}/audience/language.md` — конкретные слова/формулировки ЦА
- `projects/{{PROJECT_NAME}}/brand/positioning.md` — позиционирование и УТП (обновляет черновик)

**Оффер / прайс →**
- `projects/{{PROJECT_NAME}}/products/core-offer.md` — обновить на основе черновика: Value Stack, цена, гарантии, бонусы, условия, дедлайны, длительность
- `projects/{{PROJECT_NAME}}/products/ladder.md` — если прослеживается лестница продуктов
- `projects/{{PROJECT_NAME}}/products/lead-magnet.md` и `products/tripwire.md` — если упоминаются

**Посты / публикации →**
- `projects/{{PROJECT_NAME}}/brand/tov.md` — tone of voice: ритм, запретные слова, фирменные конструкции, уровень формальности
- `projects/{{PROJECT_NAME}}/brand/voiceprint.md` — расширить черновик до рабочего файла: 15-30 цитат, паттерны построения фраз, эмоциональные триггеры
- `projects/{{PROJECT_NAME}}/knowledge/content-bank.md` — темы, углы, форматы, которые уже работают

**Кейсы / отзывы →**
- `projects/{{PROJECT_NAME}}/knowledge/cases.md` — каждый кейс с цифрами: откуда пришёл, боль, путь, результат в числах, срок
- Пополнить `audience/pains.md` и `audience/language.md` настоящими словами пользователей

**План / стратегия / цели →**
- Обновить `projects/{{PROJECT_NAME}}/goals-extracted.md` — все цели с цифрами, сроками, приоритетами
- Пополнить `projects/{{PROJECT_NAME}}/financial/revenue-map.md` и `financial/plan.md` если есть финансовые цели

### 3. Помечать гипотезы и синтетическое
Всё, что не прямая цитата или факт из документа — `[ГИПОТЕЗА]` или `[СИНТЕЗ из X+Y]`. Особенно аккуратно с цифрами. Если пользователь сказал "примерно 500", нельзя писать "500 точно".

### 4. Зафиксировать противоречия
Секция 6 в `diagnostic-card.md` — "Противоречия в данных". Формат: "В X сказано А, в Y сказано B" — с именами источников. Это вопросы для DISCOVERY.

Типовые противоречия:
- На лендинге одна ЦА, в постах другая
- Оффер обещает X недель, в кейсах срок другой
- Pre-work говорит "нет команды", в доках фигурируют имена сотрудников
- Цена на лендинге и в оффере отличается

### 5. Классифицировать основную слабую сторону
В `diagnostic-card.md` секция 3. Выбрать ОСНОВНУЮ слабую сторону из 10 типов (полный список — в `core.md`):
🚰 Трафик / 💎 Оффер / 📉 Воронка / 📝 Контент / 🎯 Позиционирование / 🧩 Организация / 👥 Команда / 💰 Финансы / 🧠 Личные блокеры / 🔋 Энергия/фокус

Плюс 1-2 вторичных. Для основной — 3 обоснования с конкретикой из документов и pre-work. Эта классификация — отправная точка для IDEAS в `/strategist-roadmap`.

Правила:
- Если на этом этапе гипотез много и классификация неуверенная — выбрать наиболее вероятный тип и пометить раздел `[ГИПОТЕЗА, уточнить в DISCOVERY]`.
- Не бояться менять классификацию после DISCOVERY — это нормально.

### 6. Обновить CLAUDE.md проекта
В `projects/{{PROJECT_NAME}}/CLAUDE.md` поднять заполненность до 60-70%: основной оффер, ЦА (основной сегмент), слабая сторона, активы. Статус — "после UNPACK, ждёт DISCOVERY".

### 7. Краткий отчёт пользователю
Живое сообщение по схеме (формулировать заново под пользователя, без шаблонов):
- 3-5 главных тезисов распаковки (что увидел сверх первичного обсуждения)
- Нашёл противоречия: перечислить 1-3 самых важных
- Предварительная гипотеза по слабой стороне: "похоже на [тип], потому что X, Y, Z"
- Предложить `/strategist-discovery` для уточняющих вопросов по 4 блокам

## Fallback-логика
- **INTAKE не запускался** — сначала запустить его, иначе черновиков нет и нечего углублять.
- **Документов мало** — углублять то, что есть, в остальных артефактах ставить `[ДАННЫХ НЕТ, спросить в DISCOVERY]`.
- **Противоречий нет** — так и написать в секции 6: "противоречий на этом этапе не найдено". Не придумывать.
- **Слабую сторону определить нельзя** — выбрать топ-2 гипотезы, обе пометить `[ГИПОТЕЗА]`, форсировать определение в DISCOVERY блок 1.

## Выход
- Рабочие файлы в `audience/`, `products/`, `brand/`, `knowledge/`, `financial/` (где данные позволили)
- `diagnostic-card.md` заполнена на 70-80%, включая классификацию слабой стороны и противоречия
- `CLAUDE.md` проекта — статус "после UNPACK"
- В чат — 3-5 тезисов + противоречия + гипотеза слабой стороны + предложение `/strategist-discovery`
