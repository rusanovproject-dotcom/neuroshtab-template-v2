# /segments-awareness — Скилл №3 пайплайна распаковки ЦА

> Это **завершающий** скилл пайплайна. На входе — досье сегмента с заполненными секциями 0/1/2/5/6 от Скилла №2. На выходе — заполненные секции 3, 3.5, 4, 7, 8 в dossier плюс готовый `base.md`, пересобранный по правилам `_pipeline/derive-base-rules.md`.
>
> Скилл состоит из **7 шагов**. Шаг 7 — это **встроенная** реализация derive-base. Отдельного скилла `/derive-base` в пайплайне нет — правила реализуются прямо здесь, и та же логика переиспользуется в `/revise-segment` при пятничной ревизии.
>
> **Правило одного сегмента.** Скилл запускается **по одному** сегменту за раз: `/segments-awareness h-osnovatel-agentstva`, потом — отдельным заходом — `/segments-awareness w-marketolog`, потом `/segments-awareness c-konsultant`. Не три параллельно. Каждый сегмент — 1.5–2 часа фокусной работы.

---

## 1. Цель и когда запускать

### 1.1 Цель

Получить три артефакта по конкретному сегменту:

1. **Заполненная секция 3 dossier** — все 4 уровня осознанности с дословными цитатами и % концентрации.
2. **Заполненная секция 3.5** — доминантный уровень, точка входа воронки, длина воронки, стратегия.
3. **Заполненная секция 4 dossier** — lingvo-мап по 4 явным правилам (якоря / триггеры / метафоры / антипаттерны).
4. **Заполненная секция 7 dossier** — линковка с `funnel/{slug}/`.
5. **Заполненная секция 8 dossier** — anti-сегмент с цитатой отказника.
6. **Сгенерированный `{slug}/base.md`** — derived view, собранный Шагом 7 по правилам `_pipeline/derive-base-rules.md`, с шапкой `derived_from: dossier.md@v{N}`.
7. **Артефакт ритуала** — `_pipeline/awareness-workspace-{slug}.md` со всей разметкой тезисов по вёдрам (это рабочий буфер, не финальный документ).

### 1.2 Когда запускать

- После Скилла №2 (`/segments-unpack {slug}`) — секции 0, 1, 2, 5, 6 dossier заполнены.
- Перед заполнением секции 3 dossier при добавлении нового сегмента вне пайплайна (тогда вход — собранный вручную dossier).
- На квартальной ревизии когда нужно перепроверить точку входа (могла сместиться).
- При смене температуры сегмента (warm → hot или наоборот) — точка входа точно изменилась.

### 1.3 Когда НЕ запускать

- Скилл №2 ещё не прошёл (нет данных интервью в секции 2 dossier) — иди в `/segments-unpack` сначала.
- Меняем одну боль / одну цитату — это `/revise-segment`, не полный awareness-проход.
- Хочешь покрыть все 3 сегмента сразу — нет. Один сегмент за раз. Объясни клиенту почему и запусти первый.

## 2. Контракт на вход (план 7.2)

Скилл проверяет вход **до** Шага 1. Если что-то не выполнено — стоп, не лезем в работу с грязными данными.

### 2.1 Что ждём в `audience/segments/{slug}/dossier.md`

- **Секция 0** — Профили 2 представителей (имена, цитаты, контекст жизни).
- **Секция 1** — Первичные гипотезы БПСВ(К): 1.1 Соцдем, 1.2 Боли (≥15), 1.3 Потребности (≥10), 1.4 Страхи (Ты + Я раздельно), 1.5 Возражения (5+5+5).
- **Секция 2** — 7 блоков заполнены (барьеры / отношение к теме / проблемы / текущие решения / опыт платных решений / мотиваторы-демотиваторы / желаемый контент).
- **Секция 5** — Веса и приоритизация (матрица боль → LTV-импакт + матрица возражение → частота).
- **Секция 6** — Матрица «боль → артефакт воронки».
- **Шапка YAML** — `slug`, `temperature`, `status: active`, `mode: {established | early-stage | greenfield}`, `version: N`.

### 2.2 Что ждём в `audience/segments/{slug}/stream.md`

- Файл существует, не пустой.
- Минимум 5 записей с тегом `#цитата`.
- Если был запущен `/custdev` — `synthetic-interviews.md` присутствует с пометкой `[синт]`.

### 2.3 Что ждём в смежных папках

- `intel/competitors-{slug}/local/voc.md` — язык конкурентов (для Шага 4.4 антипаттерны). Если нет — Шаг 4.4 в режиме `[skipped: нет данных конкурентной разведки]`, скилл не падает, но в base оставляет пометку.
- `office/client-profile.md` или `product/` — продуктовые формулировки (для Шага 4.4).

### 2.4 Жёсткое правило валидации входа

**Скилл НЕ запускается** пока секции 0/1/2/5/6 dossier не заполнены полностью. Проверка:

```bash
bash _pipeline/_scripts/validate-dossier.sh audience/segments/{slug}/dossier.md
```

Если exit code ≠ 0 — выводи stdout валидатора клиенту дословной строкой, добавь:

> *«Не могу запустить awareness-проход — Скилл №2 (`/segments-unpack`) не докрутил досье. Чего не хватает — выше. Возвращаемся в `/segments-unpack {slug}`, потом сразу сюда.»*

И стоп. Никаких «попробую разобрать с тем что есть» — это путь в фантазию.

Если скрипта `validate-dossier.sh` нет в проекте (старая версия пайплайна) — проводи ручной чек по списку 2.1–2.3 выше, при провале — то же сообщение.

## 4. Семь шагов

### 4.1 Шаг 1 — Ритуал Ctrl+X

**Источник методологии:** `_pipeline/awareness-distribution.md` (читай его перед стартом).

**Что делаешь:**

1. Создай `_pipeline/awareness-workspace-{slug}.md` со скелетом из 5 пустых вёдер: `## L1 — Indifference`, `## L2 — Awareness`, `## L3 — Consideration`, `## L4 — Choice`, `## Не классифицировано`. Каждое ведро — заголовок + пустая строка под буллеты `- "..." — Имя`. Шаблон полностью описан в `_pipeline/awareness-distribution.md` шаги 2–3.

2. Собери все тезисы. Источники в порядке приоритета:
   - `dossier.md` секция 2 (все 7 блоков 2.1–2.7) — это полные дословные цитаты с интервью.
   - `stream.md` все записи с тегом `#цитата` за всё время.
   - `synthetic-interviews.md` если присутствует — цитаты помечай `[синт]` в скобках после атрибуции.
   - `clients/{любой клиент этого сегмента}/encounters/*.md` — если есть прямые цитаты с созвонов.

3. **Каждый осмысленный тезис** (одно предложение или короткая фраза с законченной мыслью) — Ctrl+X из источника, Ctrl+V в одно из 5 вёдер. Атрибуция обязательна: `> "{цитата}" — Имя`.

4. **Сигналы для классификации** (расширение из awareness-distribution.md):

   | Сигнал в речи | Куда |
   |---|---|
   | «не про меня», «не интересно», «и так норм», «не думал» | L1 |
   | «слышал», «читаю», «надо разобраться», «что это вообще такое» | L2 |
   | «а мне подойдёт?», «что я получу?», «как это будет у меня», «смогу ли», «подходит ли» | L3 |
   | «почему вы», «у вас есть кейсы как мой?», «сравниваю с X», «точно ли у этого продавца» | L4 |
   | Всё что между или непонятно | Не классифицировано |

5. **Не уверен?** → ВСЕГДА в «Не классифицировано». Не лей в случайное ведро. Лучше явно отложить на чек-поинт чем замусорить статистику.

6. **Один тезис = одно ведро.** Не дублируй одну цитату в два уровня. Если тезис «между двумя» — ставь на нижний (консервативная оценка).

**Ёмкость и куда что записывается.** В `awareness-workspace-{slug}.md` идёт **полная классификация всех тезисов** (может быть 50–200 цитат — это нормально, рабочий буфер не имеет hard cap). В **dossier секции 3.1–3.4 переносятся 2–3 якорные цитаты на уровень** — самые яркие, репрезентативные для каждого ведра. Это снимает противоречие: рабочий буфер растёт сколько надо, dossier остаётся ≤500 строк.

**Quality gate Шага 1:**
- [ ] Все тезисы из секции 2 dossier попали в одно из 5 вёдер (потеря тезисов = баг).
- [ ] У каждой цитаты есть атрибуция (имя + опционально дата).
- [ ] «Не классифицировано» — не помойка, там тезисы которые **реально** между уровнями.

---

### 4.2 Шаг 2 — Подсчёт концентрации

Когда вёдра заполнены — посчитай:

```
Концентрация L_N = (число тезисов в L_N / общее число классифицированных тезисов) × 100%
```

«Общее число классифицированных» = L1 + L2 + L3 + L4 (без «Не классифицировано»). Если «Не классифицировано» > 10% от общего — возвращайся на Шаг 1 с лучшими сигналами.

**Доминантный уровень** = ведро с самой высокой концентрацией. Это **точка входа воронки** — куда «приземляется» большинство людей сегмента.

**Длина воронки** = от доминантного уровня до L4 (где покупка):
- Доминанта L1 → длина 4 шага (cold-сегмент).
- Доминанта L2 → длина 3 шага (warm-сегмент).
- Доминанта L3 → длина 2 шага (hot-сегмент).
- Доминанта L4 → длина 1 шаг (super-hot, редкость).

Запиши результат в верх workspace-файла блоком `## Подсчёт концентрации ({date})` с буллетами по уровням (`L1: N/total = X%`), пометкой ← доминанта на нужной строке, отдельной строкой про «Не классифицировано» с долей, и итоговыми буллетами: Доминанта / Точка входа / Длина воронки / Тип сегмента (hot/warm/cold).

**Quality gate Шага 2:**
- [ ] Сумма L1+L2+L3+L4 = 100% ±2% (на округления).
- [ ] «Не классифицировано» ≤10% от общего числа тезисов.
- [ ] Доминантный уровень определён фактически (не «по чуйке»).

### 4.3 Шаг 3 — Самопроверка скилла + human checkpoint (без reviewer-субагента)

Это **первый stop+wait чек-поинт** скилла.

**4.3.1 Что проверяешь сам, до показа клиенту:**

1. **Тема vs Продукт.** Пробеги по L1 и L2 — нет ли там цитат с упоминанием тарифа, цены, slug продукта (например, «90К», «База 90К», «Личка 250К», конкретное название курса). Если есть — это смешение слоёв, тезис должен быть в L3 или L4. Возврат на Шаг 1 для переразметки.

   ```bash
   grep -A30 -E '^## L1|^## L2' _pipeline/awareness-workspace-{slug}.md | grep -E '[0-9]+К|тариф|База 90К|Личка 250К' && echo "FAIL: смешение слоёв L1-L2" || echo "OK"
   ```

2. **Минимум тезисов.** В каждом ведре L1–L4 минимум 3 тезиса. Если меньше — ведро не даёт статистики (для hot-сегмента L1 может быть 0 — это ок, явно отметь). Если в L3 у hot-сегмента всего 2 тезиса — данных недостаточно, возвращайся к Скиллу №2 за дополнительными интервью.

3. **Согласованность с базовой методикой.** Если у hot-сегмента доминанта L1 — что-то не так, либо сегмент классифицирован как hot ошибочно (фактически он cold), либо Шаг 1 пошёл криво. Озвучь клиенту явно.

**4.3.2 Что показываешь клиенту (формат checkpoint-диалога):**

> *«Прошёл ритуал Ctrl+X по {slug}. Разложил {N} тезисов по 4 уровням, {M} ушло в "не классифицировано". Вот картина:*
>
> *— L1 (безразличие к теме): {%} — {1–2 примера цитат}*
> *— L2 (сбор инфы о теме): {%} — {1–2 примера}*
> *— L3 (примерка продукта): {%} — {1–2 примера}*
> *— L4 (выбор поставщика): {%} — {1–2 примера}*
>
> *Доминанта — L{N} ({название}). Точка входа воронки — L{N}. Длина — {N} шагов.*
>
> *Что хочу сверить с тобой:*
>
> *1. Видишь цитату которая лежит не в том ведре? Назови — переразмечу.*
> *2. Доминанта L{N} ложится на твоё ощущение сегмента или нет? (если ты ждал L{другой} — это сигнал что либо данные кривые, либо твоя гипотеза о температуре сегмента)*
> *3. Можно идти дальше — собирать lingvo-мап?»*

**Если клиент видит расхождение** — он указывает на конкретную цитату, ты переразмечаешь в workspace-файле и пересчитываешь концентрацию (Шаг 2 заново). Возвращайся к чек-поинту.

**Если клиент апрувит** — идём на Шаг 4.

**Reviewer-субагент здесь НЕ запускается.** Намеренно. Скилл №3 — завершающий, ревью ТОП-3 уже было в Скилле №1 на чек-поинте 4. Дублировать reviewer на этапе разметки тезисов — overkill. Чек-поинт с человеком закрывает риск кривой классификации.

**Промпт для Task tool:** не используется. Весь Шаг 3 — внутри основного контекста скилла + диалог с клиентом.

### 4.4 Шаг 4 — Lingvo-мап по 4 явным правилам

Заполняешь dossier секцию 4 (`#lingvo`) **по 4 правилам Р5** из `UNPACK-ARCHITECTURE-PLAN.md`. Каждое правило тестируется отдельно (T3.7-rule-4.1, 4.2, 4.3, 4.4).

#### 4.4.1 Правило 4.1 — Живые цитаты-якоря

**Критерий:** цитата идёт в lingvo 4.1 если её **смысл** повторяется у **≥2 разных респондентов**.

**Алгоритм:**

1. Выпиши все цитаты из секций 2.x dossier и из вёдер L1–L4 workspace.
2. Группируй семантически — близкая формулировка одной мысли. Пример: «я не понимаю с чего начать» (Иван) и «я хочу чтобы кто-то провёл за руку» (Анна) — это одна группа «запрос на сопровождение».
3. Оставляй группы где **минимум 2 разных автора** (не 2 цитаты одного человека на разных созвонах — а именно разные респонденты).
4. На выходе — **топ 3–5 групп**, в каждой ОДНА каноническая цитата (самая ёмкая) + атрибуция.

**Формат dossier 4.1:** строка-цитата `> "..." — Имя` + следом курсивная строка `> *Якорь: {суть}. Подтверждение у {Имя1} и {Имя2}.*`. Эталон — `h-example-segment/dossier.md` секция 4.1.

**Failure mode:** якорь от одного человека = это не якорь. В lingvo не идёт.

#### 4.4.2 Правило 4.2 — Слова-триггеры

**Критерий:** слово (существительное / глагол / прилагательное) встречается **≥3 раз** в цитатах **разных** респондентов.

**Алгоритм:**

1. Построй частотный словарь по всем цитатам секций 2.x и вёдер L1–L4. Считай по корням (не «застрял» и «застрял-а» как разные).
2. Исключи стоп-слова: «есть», «быть», «делать», «иметь», «свой», «который», «когда», «потому что», предлоги, союзы. Если сомневаешься — исключай.
3. Для каждого оставшегося слова считай **уникальных авторов** (не упоминаний). Слово 5 раз у одного человека = 1 для критерия. Слово 3 раза у трёх разных людей = 3 для критерия.
4. Оставь топ-N с количеством уникальных авторов **≥3**.

**Формат dossier 4.2:** буллеты `- **«слово»** — суть (Имя1, Имя2, Имя3). Контекст: что значит для клиента.` Эталон — `h-example-segment/dossier.md` секция 4.2.

**Failure mode:** слово многократно у одного человека — речевая особенность, не паттерн сегмента. Не в триггеры.

#### 4.4.3 Правило 4.3 — Метафоры клиента

**Критерий:** образное сравнение с **явным маркером**: «как X», «словно X», «будто X», «это как X», «X — это для меня Y», «{что-то} съедает», «{что-то} тонет».

**Алгоритм:**

1. Grep по всем цитатам:
   ```bash
   grep -E '(как |словно |будто |это для меня |это как )' awareness-workspace-{slug}.md
   ```
2. Также grep по фразам с глагольными метафорами: «съедает», «тонет», «давит», «выжимает», «ломает».
3. Выписывай уникальные образы. Если две цитаты — про одну и ту же метафору («операционка съедает» и «операционка ест» — одно), оставляй одну.

**Формат dossier 4.3:** буллеты `- «{метафора}» — {что описывает} (Имя)`. Эталон — `h-example-segment/dossier.md` секция 4.3.

**Failure mode:** «суждения» («это сложно», «это важно») вместо образных сравнений — не метафоры. Выкини.

#### 4.4.4 Правило 4.4 — Антипаттерны (cross-check с конкурентами и продуктом)

**Критерий:** термин встречается в `intel/competitors-{slug}/local/voc.md` (язык конкурентов) **или** в `office/client-profile.md` / `product/` (наши продуктовые формулировки), **но НЕ встречается** в цитатах клиентов из секций 2.x dossier и вёдер L1–L4.

**Алгоритм:**

1. Собери список терминов-кандидатов из двух источников:
   - `intel/competitors-{slug}/local/voc.md` секции про лексику конкурентов и экспертов.
   - `office/client-profile.md` + `product/` файлы — продуктовые термины (имена тарифов, технические слова).
2. Для каждого кандидата — два grep'а:
   - **Должен быть** в источниках конкурентов / продукта (это критерий «термин существует в нашем поле»).
   - **НЕ должен быть** в `awareness-workspace-{slug}.md` и в dossier секциях 2.x (это критерий «клиенты так не говорят»).
3. Если оба условия выполнены — термин идёт в антипаттерны.

**Формат dossier 4.4:** буллеты `- ❌ «термин» — встречается у {где}, у клиентов 0 упоминаний. {Почему звучит как «не клиент»}.` Финальный буллет `- ✅ Альтернативы которые ИСПОЛЬЗУЮТ клиенты: «...», «...», «...».` Эталон — `h-example-segment/dossier.md` секция 4.4.

**Failure mode:** в антипаттернах — слова которые клиенты тоже говорят, или которых нет в источниках конкурентов / продукта. Значит придумал, не проверив. Откат.

**Если `intel/competitors-{slug}/local/voc.md` не существует** — Шаг 4.4 в режиме `[skipped: нет данных конкурентной разведки. Запусти /competitors-research]`. Пропуск явный, не молчаливый. На ревизии можно будет дозаполнить.

**Quality gate Шага 4:**
- [ ] 4.1 — каждая цитата-якорь подтверждена ≥2 разными авторами.
- [ ] 4.2 — каждое слово-триггер у ≥3 разных авторов, стоп-слова отсутствуют.
- [ ] 4.3 — каждая метафора с явным маркером.
- [ ] 4.4 — каждый антипаттерн прошёл cross-check (есть у конкурентов / продукта, нет у клиентов).

### 4.5 Шаг 5 — Заполнение секций 3 и 3.5 dossier

Из workspace-файла переносишь **2–3 якорные цитаты на уровень** в dossier секцию 3 + проценты концентрации.

**Формат dossier секции 3.** Заголовок `### Уровень N — {название}` для каждого из 4, под ним — `**Цитаты:**` 2–3 буллета `- "..." — Имя`, потом `**Концентрация:** N%`. Доминантный уровень помечается `← ДОМИНАНТНЫЙ` в заголовке. Эталон — `h-example-segment/dossier.md` секции 3.1–3.4.

**Формат секции 3.5:** четыре буллета — Доминантный уровень, Точка входа воронки, Длина воронки, Стратегия (1–2 строки под сегмент).

**Стратегия по типу сегмента (из awareness-distribution.md Шаг 6):**

- **Hot (доминанта L3):** мотивация (что изменит в жизни) + доказательства (кейсы / цифры / гарантии) + оффер.
- **Warm (доминанта L2):** ликбез по теме + мотивация + доказательства + оффер. Welcome-цепочка обязательна.
- **Cold (доминанта L1):** разбудить интерес к теме (через смежные боли) + ликбез + мотивация + доказательства + оффер. Длинный прогрев.

Стратегию пиши **под конкретный сегмент**, не шаблонно. Зацепи цитаты-якоря из 4.1.

Чек-поинты с человеком (#1 и #2 в разделе 5) закрывают риск кривой классификации и недостоверного lingvo-мапа. Этого достаточно.

**Если возникает соблазн запустить reviewer** (например, клиент пишет «прогони через критика» или есть желание подстраховать lingvo) — стоп. Это нарушение протокола скилла. Реальная альтернатива: запусти **дополнительный stop+wait чек-поинт** с конкретными вопросами клиенту, не делегируй ревью субагенту.

**Единственное место** где Task tool используется в Скилле №3 — это Шаг 4.4 при необходимости параллельного grep'а по большим объёмам (если `intel/competitors-{slug}/local/voc.md` огромный и нужно искать антипаттерны параллельно). Это техническая утилита, не reviewer.

## 9. Anti-pattern (что НЕ делать в этом скилле)

- ❌ Бросаться сразу в Шаг 4 lingvo, не пройдя Ctrl+X. Без вёдер L1–L4 ты не знаешь язык каждого уровня.
- ❌ Запускать одновременно для трёх сегментов. Контексты слипнутся, цитаты перепутаются. Один сегмент за раз.
- ❌ Делать lingvo «из головы», без cross-check по правилам Р5 4.1–4.4. Это путь в фантазию маркетолога.
- ❌ Записывать base.md если хотя бы один quality gate Шага 7 провалился. Лучше показать клиенту проблему чем закоммитить кривой derived view.
- ❌ Запускать reviewer-субагент. План явно запрещает.
- ❌ Молча skip'ать Шаг 4.4 если нет конкурентной разведки. Skip обязан быть **явным** — пометка `[skipped: причина]` в dossier 4.4 + предупреждение в чек-поинт #3.
- ❌ Резать «Не классифицировано» в случайные вёдра ради красивых процентов. Лучше явная пометка чем потеря целостности.
- ❌ Перефразировать цитаты в lingvo. Только дословно из источника. Если клиент перефразировал — попроси оригинал.
- ❌ Заполнять base.md руками. Только через Шаг 7. На пятничной ревизии — только через `/revise-segment`.
