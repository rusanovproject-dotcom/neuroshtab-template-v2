# JTBD-анализ для упаковщика

## Назначение

Генерирует гипотезы сегментов и работ (Jobs To Be Done) для B2C-продукта на основе фактуры от клиента. Методология — Advanced JTBD-методология. Скилл №2 в цепочке упаковщика. Выход этого скилла = входные данные для всех последующих (описание продукта, смыслы, оффер, лендинг).

## Что нужно на входе

**При старте — посмотри что лежит в проекте (view на папку).**
- **Есть файлы** → читай их, НЕ спрашивай пользователя «что у тебя есть»
- **Нет файлов** → спроси у пользователя что за продукт и попроси прикрепить материалы

### Минимум (можно начать с этого):
- Описание продукта — что продаёте, кому примерно, за сколько

### Желательно (повышает точность):
- **Кто уже покупает** — реальные клиенты = реальные работы. Любые данные от клиентов и о клиентах: диагностики, кейсы, отзывы, анкеты, переписки
- **Записи разговоров с клиентами / переписки менеджеров** — оттуда вытаскиваются настоящие формулировки на языке клиента
- **Конкуренты** (сайты, ленды) — чтобы понять, с какими решениями вы конкурируете в голове клиента
- **Распаковка эксперта** — если делали раньше, помогает связать сильные стороны эксперта с работами сегментов

Ничего из дополнительного не обязательно. Скилл начнёт работать с тем, что есть, и подскажет, что можно добавить потом.

## Чего скилл НЕ делает

- НЕ делает классическое описание ЦА (демография, аватары, персоны) — в JTBD это ошибка
- НЕ генерирует «возражения» в классическом смысле — вместо этого описывает барьеры переключения и проблемы текущего решения
- НЕ заменяет реальные JTBD-интервью — всё что генерирует AI = гипотезы 🔴 AI-генерация
- НЕ работает для дофаминовых продуктов (игры, соцсети, гемблинг) и неосознаваемых (еда, одежда)

## Правило коммуникации (КРИТИЧЕСКИ ВАЖНО)

Пользователь НЕ знает методологию JTBD. Он не знает что такое «Big Job», «Core Job», «Consideration Set».

**ЧТО ПОКАЗЫВАТЬ пользователю — только результат:**
- Что сейчас делаем и зачем (1-2 предложения, без терминов)
- Результат (формулировка, блок, таблица)
- Как проверить (простой тест)
- Вопрос: «Похоже на правду? Если нет — скажи что не так»

**ЧТО НЕ ПОКАЗЫВАТЬ — внутренний процесс:**
- Номера шагов алгоритма («Шаг 1», «Шаг 2»)
- Таблицы самопроверки (проверяй молча, покажи только если нашёл проблему)
- Промежуточные рассуждения
- Термины JTBD без объяснения
- **Феминитивы:** никаких «клиентки», «клиенток», «ученицы». Только «клиент», «клиенты» — независимо от пола. Гендера в анализе нет.
- **Грамотный русский язык:** перечитывай каждую формулировку перед показом. «Уйти на удалённую профессию», «зарабатывать на себя» — это тарабарщина, а не русский язык. Есть «работать на себя» и есть «зарабатывать» — это разные слова с разным значением. Не склеивай их. Если формулировка звучит криво — перепиши нормально.

ИСКЛЮЧЕНИЕ: если пользователь сам спрашивает про методологию — покажи. По умолчанию — только результат.

### Словарь замен терминов

| Термин JTBD | Что говорить пользователю |
|---|---|
| Big Job сегмента | «Главная задача этой группы клиентов» |
| Core Job | «То, что твой продукт реально делает для клиента» |
| jobStory | «Описание типичного клиента: кто он, что случилось, чего хочет» |
| Consideration Set | «С чем ты конкурируешь в голове клиента» |
| Граф работ | «Путь клиента от "хочу" до "получил"» |
| КПР | «Критические шаги — без них результат невозможен» |
| Entry Job | «Через что клиент впервые с тобой сталкивается» |
| Monetization Job | «На чём ты зарабатываешь» |
| Барьеры переключения | «Почему клиент НЕ покупает, даже если недоволен» |
| Механики ценности | «За счёт чего твой продукт лучше» |
| Налоговые работы | «Лишние действия из-за косяков текущего решения» |
| Декомпрессия | не использовать вообще |

В коммуникации — ВСЕГДА правый столбец. Термин JTBD можно в скобках для продвинутых.

## После сжатия чата (compacting)

**Если чат сжался — перечитай step-файл текущего шага заново.** После compacting ты теряешь контекст, обратную связь от пользователя и «привычки» из предыдущих сообщений. Правила из step-файла — единственный надёжный источник. Перечитай файл, перечитай черновик, продолжай работу.

## TodoList (прогресс работы)

**При старте определи среду:**
- Если доступен встроенный TodoList (Cowork) → используй его, файл НЕ создавай
- Если нет (обычный проект) → создай файл прогресса как артефакт: `projects/<main>/audience/JTBD_прогресс_<main>.md`

## Черновики (промежуточные файлы)

Черновики — рабочие файлы скилла (сегменты-черновик.md, big-job-черновик.md). Они ОБЯЗАНЫ сохраняться между compacting чата.

**Куда сохранять:** в ту же папку где лежат файлы проекта. Скилл создаёт файл прямо рядом с остальными файлами пользователя. Пользователь должен мочь его открыть и проверить.

**Правила:**
- Append-only. Только дописывать, никогда не удалять и не перезаписывать
- После записи — `view` файл и убедись что данные на месте
- Каждый блок помечен (название сегмента, номер порции)

**Файл в `projects/<main>/audience/` = артефакт.** Пользователь видит его сразу в интерфейсе. При обновлении — правь через str_replace, НЕ создавай новый файл. Один файл, обновляется на месте, без дублей.

**После создания — покажи пользователю** через present_files. Скажи: «Создал файл прогресса — по нему видно на каком мы шаге. Обновляется после каждого блока.»

**Формат файла прогресса (язык пользователя, не термины JTBD):**
```
# Прогресс JTBD-анализа: [продукт]

- [completed] Диагностика и сбор фактуры
- [in_progress] Сбор фактуры из всех источников
- [ ] Группировка клиентов
- [ ] Сегмент 1: главная задача → описание клиента → проверка → точки А/Б → путь клиента → конкуренты → барьеры → точка входа → оценка
- [ ] Сегмент 2
- [ ] Сегмент 3
- [ ] Выбор фокусного сегмента
- [ ] За счёт чего продукт лучше
- [ ] Куда расти дальше
- [ ] Финализация документа
```

После КАЖДОГО завершённого шага — обновляй файл через str_replace и показывай в чате:

> «✅ Диагностика — готово
> → Сейчас: определяю группы клиентов
> Дальше: для каждой группы — главная задача, описание, путь клиента, конкуренты, оценка»

Формат: что сделано (✅), что сейчас (→), что дальше. Коротко, 3-4 строки.

## Протокол работы (защита от потерь данных)

**Скелет документа (обязательно ДО начала работы):**
Как только начинаешь генерацию — СРАЗУ создай файл `JTBD_анализ_[продукт].md` со скелетом:

```
# JTBD-анализ: [название продукта]

## Сегменты (гипотезы)
[пусто — заполняется на шаге 01]

## Сегмент 1: [название]
### Шапка сегмента + Big Job
### jobStory
### Самокритика
### Точки А / Б
### Граф работ
### Consideration Set + Проблемы текущих решений
### Барьеры переключения
### Entry Job / Monetization Job
### Оценка сегмента

## Сегмент 2: [название]
...

## Ранжирование сегментов
## Механики создания ценности
## Стратегические гипотезы
## ⚠️ Заметки для аудита
```

**Железные правила записи:**
- Генерируй ПО ОДНОМУ БЛОКУ, жди согласования
- После согласования — СРАЗУ запиши в файл (через str_replace)
- **БЕЗ ЗАПИСИ — НЕ ПЕРЕХОДИ К СЛЕДУЮЩЕМУ БЛОКУ.** Показал → получил «ок» → записал → только тогда дальше.
- Перед ЛЮБОЙ правкой — ПЕРЕЧИТАЙ текущую версию блока ИЗ ФАЙЛА. Не правь по памяти.
- Файл = единственный источник правды. Не чат, не память — файл.
- Не будь флюгером: проверяй правки пользователя через методологию. Улучшает → прими. Ухудшает → скажи «нет» и объясни.

## Последовательность шагов

**При старте — создай TodoList (план работы):**
```
- [ ] Диагностика и сбор фактуры
- [ ] Определение групп клиентов (сегменты)
- [ ] Сегмент 1: главная задача + шапка → jobStory → проверка → Точки А/Б → граф → CS + проблемы → барьеры → Entry/Mon → оценка
- [ ] Сегмент 2: (те же шаги)
- [ ] Сегмент 3: (те же шаги)
- [ ] Ранжирование сегментов
- [ ] Механики ценности
- [ ] Стратегические гипотезы
- [ ] Финализация документа
```
По мере работы обновляй статусы: текущий шаг = in_progress, завершённый = completed. Количество сегментов уточняется по ходу — обнови TodoList когда определишь сколько их.

**Порядок выполнения:**

```
1. Прочитай steps/00-entry-and-diagnostics.md → выполни
2. Прочитай steps/01a-collect-factbase.md → собери всю фактуру в черновик
3. Прочитай steps/01b-cluster-segments.md → сгруппируй → покажи → запиши
3. Для КАЖДОГО сегмента (3-5 штук) последовательно:
   **Работай с одним сегментом за раз. Не показывай шапки всех сегментов разом.**
   Перед первым сегментом скажи: «Группа 1 из [N]. Остальные — после того как закончим с этой.»
   a. Прочитай steps/02a-extract-classify.md → извлеки запросы, классифицируй
   b. Если данных не хватает → прочитай steps/02b-ask-and-check.md → задай вопросы, проверь ответы
   c. Прочитай steps/02c-formulate-bigjob.md → сформулируй Big Job + шапку → покажи → запиши
   d. Прочитай steps/03-jobstory-generate.md → сгенерируй
   c. Прочитай steps/04-jobstory-check.md → проверь → покажи ВМЕСТЕ с jobStory → запиши
   d. Прочитай steps/05-points-ab.md → выполни → покажи → запиши
   e. Прочитай steps/06-job-graph.md → выполни → покажи → запиши
   f. Прочитай steps/07a-consideration-set.md → выполни → покажи → запиши
   g. Прочитай steps/07b-problems.md → выполни → покажи → запиши
   g. Прочитай steps/08-barriers.md → выполни → покажи → запиши
   h. Прочитай steps/09-entry-monetization.md → выполни → покажи → запиши
   i. Прочитай steps/10-segment-evaluation.md → выполни → покажи → запиши
4. Прочитай steps/11-ranking.md → выполни → покажи → запиши
5. Прочитай steps/12-value-mechanics.md → выполни → покажи → запиши
6. Прочитай steps/13-strategic-hypotheses.md → выполни → покажи → запиши
7. Финализируй документ. Скажи пользователю:
   «Документ готов. Для финальной проверки — открой новый чат
    и вызови скилл jtbd-critic. Передай ему готовый документ.
    Свежий взгляд в чистом контексте найдёт то, что мы могли пропустить.»
```

## Место в цепочке

```
① Распаковка экспертности (опционально)
     ↓
② JTBD-анализ ← ЭТОТ СКИЛЛ
     ↓
④ Описание продукта → ⑤ Маркетинговые смыслы → ⑥ Оффер → ⑦ ToV → ⑩ Лендинг
```

## Формат выхода

Документ с 7 блоками:

1. 3-5 сегментов (для каждого: шапка + Big Job сегмента, jobStory, самокритика, Точки А/Б, граф работ с КПР, Consideration Set + проблемы текущих решений с налоговыми работами, барьеры, Entry/Monetization, оценка)
2. Ранжирование сегментов
3. Механики создания ценности (2-3 для фокусного сегмента)
4. Стратегические гипотезы
5. Разметка достоверности (сводная)
6. ⚠️ Заметки для аудита
