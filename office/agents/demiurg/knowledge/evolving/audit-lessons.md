## Ошибка 1: КЛОНИРОВАНИЕ (все трое отметили)

**Что было:** Агенты создавались как клоны шаблона — одинаковый identity,
одинаковая структура, разные только имена. "Ты — CMO. Отвечаешь за маркетинг."

**Почему плохо:** Агент-клон не знает свой домен. Он общий = бесполезный.
Настоящий CMO знает tone of voice, anti-слова, формулы хуков, ЦА с цитатами.

**Как предотвратить:**
1. ПЕРЕД сборкой агента — собери DATA PACK (knowledge/architecture/data-packs.md)
2. Identity должен содержать УНИКАЛЬНЫЙ принцип домена, не общие слова
3. После сборки — "тест уникальности": сравни identity с другими агентами.
   Если можно заменить имя и смысл не изменится — это клон. Переписать.
4. Каждый агент отвечает на вопрос: "Чем я ОТЛИЧАЮСЬ от других агентов?"

**Чеклист (вставлять в голову при создании каждого агента):**
- [ ] Есть ли у агента СВОЙ принцип? (не "делай хорошо", а конкретный)
- [ ] Есть ли domain-specific правила? (не общие, а из предметной области)
- [ ] Есть ли свой тон/стиль? (не одинаковый для всех)
- [ ] Может ли другой агент делать то же самое? (если да — зачем этот?)

## Ошибка 2: ДУБЛИРОВАНИЕ ФАЙЛОВ (внешний аудит выявил)

**Что было:** 10 файлов-дублей между architect/ и demiurg/. References/ копировали
knowledge/. При обновлении одного — второй устаревал.

**Почему плохо:** "Дубль = баг" — собственный принцип Demiurg, который он нарушал.

**Как предотвратить:**
1. НИКОГДА не копируй файл — ссылайся. Handle-паттерн обязателен.
2. Перед создданием файла: `grep` ключевые фразы по всей системе.
   Если нашёл похожий файл — ссылайся на него, не создавай копию.
3. После сборки: validate.sh проверяет дубли автоматически.
4. Одна папка knowledge/ для всех. Агенты ССЫЛАЮТСЯ, не КОПИРУЮТ.

## Ошибка 3: ПРОПУСК ВАЛИДАЦИИ (внешний аудит, ось «Инструменты»)

**Что было:** scaffold.sh создавал файлы, но никто не проверял результат.
Broken links, пустые INDEX.md, TODO-заглушки оставались незамеченными.

**Почему плохо:** Агент с broken links не может найти свои знания.
INDEX с TODO — навигация не работает. Пользователь получает сломанную систему.

**Как предотвратить:**
1. ПОСЛЕ каждой сборки — `bash scripts/validate.sh <target>`. ОБЯЗАТЕЛЬНО.
2. Если validate FAIL — НЕ показывать пользователю. Исправить сначала.
3. validate.sh проверяет: ссылки, INDEX, размеры, TODO, дубли, пустые файлы.
4. Micro-validation после каждого агента (не ждать до конца).

## Ошибка 4: ИЗОЛЯЦИЯ ОТ СИСТЕМЫ (внешний аудит, ось «Координация»)

**Что было:** Demiurg создавал агентов, но не связывал их с командой.
Не обновлял AGENTS.md, context.md, ops/log.md. Не передавал на ревью.

**Почему плохо:** Агент без связей = изолированный сотрудник.
Никто не знает что он существует, он не знает о других.

**Как предотвратить:**
1. ПОСЛЕ сборки обязательно: обновить AGENTS.md (если есть в workspace)
2. ПОСЛЕ сборки: предложить конвейер Demiurg → независимый аудитор → полировщик
3. В CLAUDE.md каждого агента: секция "НЕ отвечает за → передай {кому}"
4. Проверить: нет ли пересечений зон с существующими агентами
5. Проверить: прописан ли reports_to (кому подчиняется)

## Ошибка 5: ПЕРЕГРУЗКА CLAUDE.md (внешний аудит, ось «Экономия контекста»)

**Что было:** CLAUDE.md содержал inline знания: принципы (8 строк), инструменты
(5 строк), guardrails (4 строки), error handling (6 строк). 123 строки.
CLAUDE.md грузится КАЖДЫЙ раз — каждая лишняя строка = лишние токены.

**Почему плохо:** 123 строки × каждое сообщение = расход контекста на правила,
а не на работу. Принципы дублировали knowledge/.

**Как предотвратить:**
1. CLAUDE.md = 60-100 строк. Больше — вынеси в knowledge/.
2. В CLAUDE.md: identity + навигация + решения. ВСЁ.
3. Правила, чеклисты, примеры → knowledge/ с handle-ссылками.
4. Тест: "Нужна ли эта строка В КАЖДОМ сообщении?" Если нет → в knowledge/.

## Ошибка 6: КОНФЛИКТ ТРИГГЕРОВ (Vibe Coder 7/10)

**Что было:** build-team и heavy-build имели одинаковые триггеры:
"пересобрать систему", "full build". Claude не мог выбрать.

**Почему плохо:** Неправильный скилл = неправильный процесс = плохой результат.

**Как предотвратить:**
1. Каждый скилл = уникальный scope. Нет пересечений.
2. ОБЯЗАТЕЛЬНО в description: "DO NOT use when..." (near-miss).
3. В CLAUDE.md агента: триггер-таблица "сигнал → скилл".
4. При создании нового скилла: grep существующие descriptions на пересечения.

## Ошибка 7: САМООЦЕНКА БЕЗ ВНЕШНЕГО РЕВЬЮ (Vibe Coder 8.5/10)

**Что было:** Demiurg сам себе ставил оценки в петле Judge-Refine.
Self-grading bias: модель завышает оценки собственной работе.

**Как предотвратить:**
1. ВСЕГДА предлагай внешний ревью после сборки:
   "Передать независимому ревьюеру для аудита?"
2. Собственный review = необходимый минимум, не финал.
3. Для критичных систем (>5 агентов): внешний ревью ОБЯЗАТЕЛЕН.

## Ошибка 8: НЕТ DATA PACK (обнаружено при текущем аудите)

**Что было:** Агент создавался с правильной структурой, но без ДАННЫХ.
Копирайтер без tone of voice. Дизайнер без брендбука. CTO без стека.

**Почему плохо:** Качество агента = качество промта × качество данных.
Промт без данных = пустышка с правильной структурой.

**Как предотвратить:**
1. ПЕРЕД сборкой: определи тип агента → загрузи data-packs.md
2. Проверь: есть ли обязательные данные для этого типа?
3. Если данных нет → собери (интервью, анализ, веб-поиск)
4. Если данные неполные → предупреди пользователя: "без X агент будет слабым"
5. Score готовности данных: <5 = сборка заблокирована

## Резюме: 8 правил для каждой сборки

1. **Не клонируй** — каждый агент уникален (тест уникальности)
2. **Не дублируй** — ссылайся, не копируй (grep перед созданием)
3. **Валидируй** — validate.sh после каждой сборки (обязательно)
4. **Связывай** — AGENTS.md, reports_to, НЕ отвечает за → кому
5. **Экономь** — CLAUDE.md 60-100 строк, остальное в knowledge/
6. **Разграничивай** — near-miss в скиллах, нет пересечений
7. **Проверяй внешне** — предлагай независимый ревьюер после сборки
8. **Корми данными** — data pack ДО сборки, не после
