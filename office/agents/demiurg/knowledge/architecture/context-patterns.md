# Паттерны контекстной загрузки

Как агенты загружают информацию эффективно, не сжигая весь контекст.

---

## 1. Handle Pattern (ссылки вместо содержимого)

Агент видит ПУТЬ к файлу и описание в 1 строку.
Загружает файл целиком только когда задача этого требует.

```markdown
## Знания (загружай по необходимости)
- `knowledge/brand/positioning.md` — позиционирование продукта
- `knowledge/audience/avatar.md` — портрет ЦА
- `knowledge/methodology/program-v2.md` — структура программы
```

**Когда использовать:** всегда. Это базовый паттерн для любого агента.

**Как работает:**
1. Агент видит список handle'ов при старте
2. Получает задачу "напиши пост про мастер-группу"
3. Решает: нужен positioning.md + program-v2.md
4. Загружает только эти 2 файла, остальные не трогает

**Антипаттерн:** копировать содержимое файлов в CLAUDE.md.
Это убивает контекст и файлы устаревают (дубль проблема).

---

## 2. Summary Pattern (большой файл + компактная выжимка)

Для файлов >200 строк: создай summary-версию ≤50 строк.
Агент читает summary, и только если нужны детали — полный файл.

```
knowledge/methodology/
├── program-v2.md           ← полный (400 строк)
└── program-v2.summary.md   ← выжимка (40 строк)
```

Или через отдельный файл:
```
context.md      ← summary всего состояния (≤50 строк)
state.md        ← полное состояние (200+ строк)
```

**Когда использовать:** когда есть файлы >200 строк которые читаются часто.

**Правила:**
- Summary обновляется при каждом изменении оригинала
- Summary содержит: ключевые факты + ссылку на полный файл
- Формат: "Подробности: `path/to/full-file.md`"

---

## 3. Specialist Pattern (каждый видит только своё)

Каждый агент имеет явный список: ЧТО читает и ЧТО НЕ читает.
Это не просто рекомендация — это жёсткое правило в CLAUDE.md агента.

```markdown
# CTO

## Мой контекст
Читаю: projects/*, ops/log.md, context.md
НЕ читаю: knowledge/brand/, knowledge/audience/
```

**Когда использовать:** когда >2 агентов и есть риск пересечений.

**Почему это важно:**
- Экономия контекста (CTO не тратит токены на маркетинг)
- Чёткие границы (нет "а может мне самому пост написать?")
- Предсказуемость (знаешь что агент точно НЕ полезет куда не надо)

---

## 4. Progressive Disclosure (3 уровня глубины)

Информация загружается послойно, от общего к частному:

**Уровень 1 — Identity (всегда в контексте)**
- CLAUDE.md агента: кто, что, принципы
- ≤300 токенов
- Загружается автоматически при каждой сессии

**Уровень 2 — Context Pack (загружается при старте задачи)**
- context.md: текущие приоритеты
- AGENTS.md: кому передать если не моё
- ≤2000 токенов суммарно
- Агент читает ЭТО в начале каждой задачи

**Уровень 3 — Knowledge Base (по запросу)**
- Файлы из knowledge/ через INDEX.md
- Детали проектов из projects/*/CLAUDE.md
- Без ограничения объёма, но загружается ТОЛЬКО когда нужно

```
Identity (300 tok) → Context Pack (2000 tok) → Knowledge (по запросу)
         ↑ всегда            ↑ при задаче           ↑ при необходимости
```

## 5. Append-Only Memory (память агентов)

Агенты записывают выводы ПОСЛЕ каждой задачи. Три типа:

**Episodic** (что произошло): факты, события, результаты
```
## 2026-03-22
- Обновил лендинг <your-app> — новый hero-блок
- Клиент Маша попросила перенести коуч-сессию на среду
```

**Semantic** (что выучили): знания, паттерны, инсайты
```
## Маркетинг
- Посты с личным опытом дают x3 вовлечённость vs экспертные
- Лучшее время постинга: 10:00 и 19:00 МСК
```

**Procedural** (как делать): процедуры, чеклисты
```
## Деплой <your-app>
1. git push origin main
2. ssh <your-server>
3. cd /opt/<your-app> && git pull && docker compose up -d
```

**Где хранить:** `ops/memory/episodic.md`, `ops/memory/semantic.md`, `ops/memory/procedural.md`

**Правило:** append-only. Не удалять старые записи. Раз в месяц — архивировать
старое в `archive/memory/`.
