# Анатомия идеального агента — эталон готового результата

Этот документ — НЕ шаблон для копирования. Это эталон: как выглядит
завершённый, полностью собранный агент на score 9+/10.

Каждый агент УНИКАЛЕН. Клон шаблона = плохой агент. Уникальность рождается
из домена, задач и связей с командой.

---

## Структура файлов готового агента

```
agent-name/
├── CLAUDE.md              ← Мозг: identity + навигация (60-150 строк)
├── skills/
│   └── main-skill/
│       ├── SKILL.md       ← Ядро скилла: триггеры + процесс (200-500 строк)
│       ├── references/    ← Тяжёлые знания (Progressive Disclosure)
│       │   ├── patterns.md
│       │   └── examples.md
│       ├── scripts/       ← Автоматизация (validate, generate)
│       │   └── check.sh
│       └── assets/        ← Шаблоны, конфиги
│           └── template.md
├── knowledge/
│   ├── INDEX.md           ← Карта знаний (обязательно если >3 файлов)
│   ├── domain/            ← Знания по предметной области
│   │   └── core.md
│   └── evolving/          ← Самообучение (что выучил, что зафейлилось)
│       └── learnings.md
└── MEMORY.md              ← Память между сессиями (опционально)
```

**Минимальный агент:** только CLAUDE.md (solo-паттерн, простые задачи)
**Типичный агент:** CLAUDE.md + 1 skill + knowledge/
**Продвинутый агент:** полная структура выше + memory + evolving/

## Слой 1: CLAUDE.md — Мозг агента

CLAUDE.md = то что агент видит ВСЕГДА. Поэтому каждая строка должна
"зарабатывать" своё место.

### Обязательные секции (в этом порядке):

```markdown
# {Имя} — {Суть в 3-5 словах}

{Identity: 2-4 предложения. Кто, что, главный принцип. НЕ перечень обязанностей.}

## Зона ответственности
Отвечаешь за: {3-5 пунктов}
НЕ отвечаешь за: {2-3 пункта → кому передать}

## Контекст
При старте загрузи: {2-3 файла}
По запросу: {файлы с условием КОГДА грузить}

## Как работать
{3-7 шагов основного процесса}

## Решения
- Сам: {примеры}
- С обсуждением: {примеры}
- Эскалация: {примеры}
```

### Что делает Identity УНИКАЛЬНЫМ (не клоном):

Плохо (клон): "Ты — CMO. Отвечаешь за маркетинг и контент."
Хорошо (уникальный): "Ты — CMO. Голос бренда: исповедальный бунт, без
корпоративщины. Принцип: каждый текст = конкретное действие читателя.
Пишешь от первого лица пользователя, не от бренда."

**Уникальность рождается из:**
1. Специфика домена (не "маркетинг", а "growth через Telegram для коучей")
2. Принцип работы (не "делай хорошо", а "рабочий код важнее красивого")
3. Тон/стиль (не "будь профессионален", а "пиши как старший коллега в Slack")
4. Конкретные ограничения домена (не "качественно", а "identity ≤ 300 tok")

### Лимиты CLAUDE.md:
- 60-150 строк (идеально 80-100)
- Identity ≤ 300 токенов
- Ссылки на файлы (handle-паттерн), не inline контент
- Максимум 5 файлов в "обязательном" контексте
- Позитивные формулировки ("делай X" а не "не делай Y")

## Принцип структурности (ОБЯЗАТЕЛЕН для каждого агента)

Каждый агент — мастер этапных систем. Не фристайл, а чёткая механика.

**Правила:**
1. **Агент всегда идёт по своим шагам.** Процесс описан в скилле, агент следует ему, не импровизирует
2. **Узкие скиллы > универсальные.** Один скилл = одна задача. "write-post" а не "create-content"
3. **Субагенты для параллельной работы.** Если задача делится на независимые части — субагенты
4. **Quality gates между этапами.** Не переходи пока предыдущий не PASS
5. **Каждый шаг = конкретный результат.** Не "подумай", а "создай файл X с содержанием Y"

**Demiurg создаёт агентов по этому принципу:**
- В CLAUDE.md: секция "Как работать" = пронумерованные шаги
- В скиллах: фазы с условиями перехода и exit criteria
- В knowledge: "когда загружать" для каждого файла (не "грузи всё сразу")
- Если задача сложная: описать где агент запускает субагентов

**Антипаттерн:** агент с одним скиллом "do-everything" на 800 строк. Разбей на 3-4 узких.

## Слой 2: Skills — Процессы агента

Скилл = управляемый процесс. Не советы, а пошаговые инструкции.

### Что делает скилл УПРАВЛЯЕМЫМ:

1. **Чёткий триггер** — description с синонимами и near-miss
2. **Фазы с quality gates** — условие перехода между шагами
3. **Примеры Input → Output** — минимум 2, включая edge case
4. **Exit criteria** — когда скилл ЗАВЕРШЁН (не "когда хорошо", а конкретно)
5. **Error recovery** — что делать если шаг провалился

### Пример управляемого скилла vs неуправляемого:

```
НЕУПРАВЛЯЕМЫЙ:
"Напиши пост. Сделай его интересным. Используй storytelling."
→ Результат: непредсказуемый, каждый раз разный

УПРАВЛЯЕМЫЙ:
Фаза 1: Прочитай knowledge/brand/tone.md → определи формат
Фаза 2: Выбери hook из references/hooks-library.md
Фаза 3: Напиши по структуре: hook → tension → insight → CTA
Фаза 4: Проверь по чеклисту: ≤2000 знаков? Есть CTA? Нет корпоративщины?
Exit: файл создан + чеклист пройден
```

## Слой 3: Knowledge — Знания агента

Knowledge = факты и паттерны, которые агент загружает по запросу.

### Принципы организации:

1. **INDEX.md** — обязателен если >3 файлов. Формат:
   ```markdown
   | Файл | Что внутри | Когда загружать |
   |------|-----------|----------------|
   | domain/core.md | Основы продукта | При любой задаче |
   | domain/audience.md | Портрет ЦА | При создании контента |
   ```

2. **Один факт = одно место** — никогда не дублируй. Ссылайся.
3. **Файл ≤ 300 строк** — если больше, разбей на 2+ файла
4. **evolving/** — самообучение. Агент записывает что выучил, что зафейлилось

### Что НЕ класть в knowledge/:
- Процессы (это skills/)
- Описание самого агента (это CLAUDE.md)
- Чужие знания (ссылайся на общий knowledge/)
- Временные данные (это context.md или state)

## Слой 4: Связи — Агент в команде

Агент не живёт в вакууме. Он часть системы.

### Обязательные связи:

1. **Вверх:** кому подчиняется, кому эскалирует
2. **В стороны:** кому передаёт задачи, от кого получает
3. **Общие точки:** context.md (читает), ops/log.md (пишет)

### Как прописать связи:

В CLAUDE.md:
```markdown
## НЕ отвечаешь за:
- Код → передай CTO (agents/cto.md)
- Дизайн → передай Designer (agents/designer.md)
```

В AGENTS.md (корневой):
```markdown
| Агент | Задачи | Передаёт |
|-------|--------|----------|
| CMO | контент, воронки | CTO (если код), Designer (если визуал) |
```

### Антипаттерн — изолированный агент:
Агент без связей = агент который будет делать всё сам.
Это ведёт к пересечениям и дублированию.

## Слой 5: Memory — Память агента (опционально)

Для агентов которые работают регулярно и должны учиться.

### Структура MEMORY.md:

```markdown
# {Agent} — Memory

## Decisions (что решил и почему)
### YYYY-MM-DD: {тема}
- What: {решение}
- Why: {причина}

## Patterns (закономерности)
- {паттерн 1}

## Failures (что НЕ сработало)
- {описание}

## Context (что помнить в следующей сессии)
- {факт}
```

## Чеклист: "Агент готов к продакшену?"

### CLAUDE.md ✓
- [ ] Identity ≤ 300 токенов, уникальный (не клон шаблона)
- [ ] Есть принцип работы (не общие слова)
- [ ] "Отвечает за" — 3-5 конкретных пунктов
- [ ] "НЕ отвечает за" — 2-3 пункта с указанием КОМУ передать
- [ ] Контекст: ≤5 обязательных файлов, остальное "по запросу"
- [ ] Handle-паттерн: ссылки, не inline контент
- [ ] Все ссылки валидны (файлы существуют)
- [ ] 60-150 строк (не больше)
- [ ] Решения: 3 уровня (сам / обсуждение / эскалация)

### Skills ✓
- [ ] Description: "pushy", с синонимами и near-miss
- [ ] Body ≤ 500 строк
- [ ] Минимум 2 примера Input → Output
- [ ] Фазы с quality gates (условие перехода)
- [ ] Exit criteria: конкретное условие завершения
- [ ] Error recovery: что делать если шаг провалился
- [ ] Heavy content → references/ (Progressive Disclosure)
- [ ] Позитивные формулировки

### Knowledge ✓
- [ ] INDEX.md с колонкой "Когда загружать"
- [ ] Один факт = одно место (нет дублей)
- [ ] Файлы ≤ 300 строк
- [ ] Нет процессов (процессы → skills/)
- [ ] evolving/ — есть место для самообучения

### Связи ✓
- [ ] Прописано кому подчиняется
- [ ] Прописано кому передаёт задачи (с путями к файлам)
- [ ] Нет пересечений зон ответственности с другими агентами
- [ ] Прописан в AGENTS.md (корневой)

### Финальная проверка ✓
- [ ] validate.sh PASS
- [ ] DRY-RUN: 2 сценария пройдены мысленно за ≤3 шага
- [ ] Агент отличается от других (уникальный identity, свой принцип)
