# Как писать скиллы, которые реально работают

## Принцип #1: Why > MUST

Claude — умная модель. Она лучше обобщает, когда понимает причину.

❌ "ВСЕГДА используй TypeScript. НИКОГДА не пиши JavaScript."
✅ "Используй TypeScript — проект типизирован, и нетипизированный код ломает
   автодополнение для всей команды. JavaScript допустим только в конфигах."

## Принцип #2: Примеры бьют правила

Несколько хороших примеров дают лучший результат, чем длинные правила.

```markdown
## Формат коммитов

Используй conventional commits:

**Пример 1:**
Input: Добавил JWT авторизацию
Output: feat(auth): implement JWT-based authentication

**Пример 2:**
Input: Починил отображение цен с копейками
Output: fix(pricing): correct decimal display in cart total
```

## Принцип #3: Lean > Fat

Каждое правило в скилле должно "зарабатывать" своё место.

**Тест:** убери правило и запусти скилл. Если результат не изменился —
правило не нужно, удали. Если ухудшился — оставь.

Практика: после 2-3 итераций тестирования, прочитай transcripts выполнения.
Если агент тратит время на действия, которые не дают результата — найди в
скилле инструкцию, которая это вызывает, и удали/перепиши.

## Принцип #4: Generalize, не overfit

Скилл будет использоваться на разных задачах, не только на тестовых примерах.
Если скилл работает идеально на 3 тестовых промтах, но сломается на 4-м —
он бесполезен.

❌ "Если пользователь просит таблицу, используй 3 колонки: Имя, Дата, Статус"
✅ "Определи из контекста задачи какие колонки нужны. Обычно 3-7 колонок
   достаточно. Больше 7 — подумай, не нужна ли группировка."

## Принцип #5: Script > Hand-work

Если все 3 тестовых запуска скилла привели к тому, что Claude написал
одинаковый helper-скрипт — значит, этот скрипт нужно положить в scripts/.

Примеры что выносить:
- Парсинг/форматирование данных
- Генерация файлов по шаблону
- Валидация результатов
- Scaffold новых структур

## Цикл разработки скилла

```
1. Захватить намерение
   ↓ Что скилл делает? Когда триггерится? Какой output?
2. Написать черновик SKILL.md
   ↓ Не идеальный — просто рабочий
3. Написать 2-3 тестовых промта
   ↓ Реалистичные, как настоящий пользователь
4. Запустить и посмотреть результат
   ↓ С baseline (без скилла) и с скиллом
5. Показать пользователю результат
   ↓ Собрать feedback: что не так?
6. Переписать скилл на основе feedback
   ↓ Обобщить замечания, не фитить под конкретный пример
7. Повторить 4-6 пока не ОК
```

## Тестирование

### Тестовые промты

Пиши как РЕАЛЬНЫЙ пользователь, а не как QA-инженер:

❌ "Создай PDF документ с таблицей из 3 колонок и графиком"
✅ "слушай у шефа завтра отчетная встреча, можешь быстро из этих цифр
   красивый pdf собрать? там продажи за квартал, помесячно"

Включай:
- Разговорный стиль, опечатки, сокращения
- Контекст (зачем нужно, для кого)
- Неточные формулировки (как реальные люди говорят)
- Разную длину: короткие и длинные запросы

### Оценка качества

**Количественная:** assertions — проверяемые утверждения
- "В результате есть график" → можно проверить программно
- "Файл создан и не пустой" → проверяемо

**Качественная:** человеческий review
- "Текст звучит естественно" → субъективно, нужен человек
- "Дизайн выглядит профессионально" → не автоматизируешь

Для субъективных скиллов (копирайтинг, дизайн) — фокус на human review.
Для объективных (парсинг, генерация файлов) — автоматические assertions.

## Оптимизация Description (Trigger Tuning)

После того как скилл работает, оптимизируй description для лучшего триггеринга:

1. Создай 20 eval-запросов: 10 should-trigger + 10 should-not-trigger
2. Should-not-trigger = near-misses (похожие, но нужен другой скилл)
3. Запусти optimization loop (если есть `claude -p`)
4. Результат: description, которое триггерится точнее

**Ключевое:** bad eval queries → bad description. Инвестируй время в качественные тесты.
