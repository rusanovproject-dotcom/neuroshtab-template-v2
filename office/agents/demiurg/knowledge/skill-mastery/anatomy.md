# Анатомия скилла — полный разбор

## Что такое скилл

Скилл = инструкция + ресурсы, которые Claude загружает по запросу.
Скилл сам ничего не делает — он учит Claude *как* делать конкретную задачу.

## Структура на диске

```
skill-name/
├── SKILL.md              ← обязательно: ядро скилла
│   ├── YAML frontmatter  ← name + description (триггер)
│   └── Markdown body     ← инструкции (≤500 строк)
├── references/           ← доки, загружаемые по ходу
│   ├── patterns.md
│   └── examples.md
├── scripts/              ← исполняемый код
│   └── generate.sh
└── assets/               ← файлы для вывода (шаблоны, шрифты, иконки)
    └── template.html
```

## Три уровня загрузки (Progressive Disclosure)

| Уровень | Что загружается | Когда | Размер |
|---------|----------------|-------|--------|
| **1. Metadata** | name + description из frontmatter | Всегда в контексте | ~100 слов |
| **2. Body** | SKILL.md целиком | Когда скилл триггерится | ≤500 строк |
| **3. Resources** | references/, scripts/ | Когда body говорит "прочитай X" | Неограничено |

Это ключевой паттерн. Скилл НЕ загружает всё сразу. body SKILL.md говорит
"когда нужна тема X — прочитай references/X.md". Так экономится контекст.

## YAML Frontmatter

```yaml
---
name: my-skill
description: >
  Описание что делает скилл и КОГДА его использовать.
  Это ЕДИНСТВЕННЫЙ механизм триггеринга.
  Пиши "pushy" — лучше перетриггерить, чем недотриггерить.
---
```

### Правила description

**Проблема #1 в скиллах: undertriggering.** Claude склонен НЕ вызывать скилл,
даже когда он подходит. Поэтому description должен быть "pushy":

❌ Плохо: "Создание дашбордов для отображения данных"
✅ Хорошо: "Создание дашбордов для отображения данных. Используй этот скилл
   когда пользователь упоминает дашборды, визуализацию, метрики, графики,
   или хочет отобразить любые данные, даже если не просит 'дашборд' явно."

**Включай в description:**
- Что скилл делает (основное)
- Когда триггерить (перечисли ситуации)
- Ключевые слова и синонимы
- Когда НЕ триггерить (если есть путаница с другими скиллами)

## Body (основные инструкции)

### Стиль написания

- **Imperative** (повелительный): "Используй", "Прочитай", "Создай"
- **Why > MUST**: объясняй ПОЧЕМУ это важно, а не просто приказывай
- **Примеры**: 2-3 хороших примера бьют 10 правил
- **Не перегружай**: если >500 строк — выноси в references/

### Структура body

```markdown
# Название скилла

{1-3 предложения: что делает, главный принцип}

## Когда использовать
{Конкретные сценарии}

## Как работать
{Пошаговые инструкции}

## Формат вывода
{Шаблон результата}

## Примеры
{Input → Output}

## Ссылки на ресурсы
- `references/X.md` — когда нужно Y
- `scripts/Z.sh` — запусти для W
```

## References (справочные доки)

Правила:
- Каждый файл — отдельная тема
- Если файл >300 строк — добавь оглавление в начало
- SKILL.md должен явно указывать КОГДА какой reference читать
- Не дублируй контент между references — ссылайся

## Scripts (исполняемый код)

Когда использовать scripts/:
- Повторяющиеся действия (создание файлов, форматирование)
- Детерминированные операции (парсинг, конвертация)
- Валидация результатов (проверка структуры)

Скрипты запускаются БЕЗ загрузки в контекст — Claude просто делает `bash scripts/X.sh`.

## Assets (файлы для вывода)

- Шаблоны документов (.html, .docx templates)
- Иконки, шрифты для генерации
- Конфиги, которые копируются в проект

## Антипаттерны

❌ Description <20 слов → недотриггерится
❌ Body >500 строк → Claude потеряет фокус
❌ Всё в одном файле → контекст выгорает на первой задаче
❌ ALWAYS/NEVER в caps → лучше объясни почему
❌ Нет примеров → Claude додумает сам (плохо)
❌ Script'ы внутри body → вынеси в scripts/, меньше шума
