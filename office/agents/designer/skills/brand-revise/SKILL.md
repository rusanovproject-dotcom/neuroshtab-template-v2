# Brand Revise — правка Brand Book

Твоя задача — аккуратно изменить существующий Brand Book проекта так, чтобы не сломать систему. Один параметр за раз, с подтверждением, с фиксацией в истории.

## Вход

- **Текущий Brand Book** — `knowledge/brand/{project}/brand-book.md`. Если файла нет — **остановись**, запусти `brand-onboarding`, это не правка а сборка с нуля.
- **Что именно меняем** — спроси пользователя одной фразой если непонятно.

## Шаги

### 1. Прочитай текущий Brand Book

Полностью. Запомни палитру, шрифты, настроение, запреты. Пригодится для блока «до/после».

### 2. Уточни что меняем (если непонятно)

Один вопрос:
> *«Заменить [шрифт заголовков / акцентный цвет / настроение] — понял. На что меняем? Предложу 3 варианта под твоё текущее настроение, или у тебя уже есть конкретика?»*

Если пользователь говорит «хочу совсем другой стиль / переделай полностью» — **не правь**, запусти `brand-onboarding` заново (это уже новый Brand Book, а не правка).

### 3. Покажи «до/после» кратко

Не заставляй пользователя копать в файл. Одним блоком:

> *«Было: Inter для заголовков.*
> *Станет: Unbounded Medium.*
> *Это сделает блоки жёстче, структурнее, добавит архитектурность в настроение.*
> *Идём?»*

Для цвета — покажи hex и словами: *«Было #E1A17A (тёплая охра). Станет #C77D50 (терракот — более земляной, плотный, менее жёлтый).»*

### 4. Правь после подтверждения

Пользователь сказал «да / идём / го» — вноси правку в `knowledge/brand/{project}/brand-book.md`. **Не меняй больше одного параметра за раз без отдельного подтверждения.**

### 5. Добавь запись в «Историю изменений»

В секции `## История изменений` таблицы — **новая строка** (append, не перезаписывай старые):

```
| 2026-04-22 | Заменил шрифт заголовков Inter → Unbounded Medium. Причина: пользователь попросил больше жёсткости и архитектурности. |
```

Формат: `YYYY-MM-DD | [что изменил глаголом] [было → стало]. Причина: [короткая, одна фраза].`

### 6. Оповести пользователя

> *«Обновил Brand Book. Следующий дизайн пойдёт с новым [шрифтом / цветом / настроением]. Хочешь сразу применить — сделаем [логичный следующий шаг]?»*

Это запуск проактивности — сразу предложи куда двинуть дальше.

## Правила

- **Один параметр за раз.** Пользователь сказал «поменяй шрифт И добавь цвет» — разбиваешь на 2 итерации, подтверждаешь отдельно.
- **Запреты не удаляй молча.** Если пользователь говорит «убери запрет на фиолетовый» — перепиши с пометкой: `~~фиолетовые градиенты~~ (убран 2026-04-22)`. Через strikethrough. Это чтобы помнить историю.
- **Если правка ломает систему** — мягко скажи. Пример: пользователь просит добавить 4-й акцентный цвет — *«Это сломает палитру, 3 акцента — уже много. Может лучше заменим один из существующих? Покажу какой.»*
- **Hex-коды не галлюцинируй.** Если пользователь говорит «хочу чуть темнее охры» — предложи 3 конкретных hex на выбор, не выдумывай «примерно такой».
- **Не трогай референсы и проект** в секциях Brand Book — только то что пользователь попросил.

## Выход

1. Обновлённый `knowledge/brand/{project}/brand-book.md` с новой записью в «Истории изменений»
2. Короткое оповещение пользователю с предложением следующего шага
