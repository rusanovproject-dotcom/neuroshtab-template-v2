# Принципы интеграции агентов в офис

Обязательный чеклист при добавлении любого нового агента в пользовательский офис. Следовать по порядку.

Архитектурная модель: базовый шаблон **минимальный** (Director + Strategist + Demiurg). Все остальные помощники поставляются **готовыми паками** в `_agent-packs/<name>/` и ставятся пользователем командой `/install-agent <name>` когда он готов.

---

## 0. Agent Packs — как поставляются новые помощники

Это основной механизм расширения офиса.

- **Пак** — автономная папка `_agent-packs/<name>/` с полным набором файлов агента плюс манифест установки.
- **Установка** — пользователь пишет `/install-agent <name>` (или живым языком: *"установи дизайнера"*, *"подключи копирайтера"*). Универсальный скилл `install-agent` читает пак и применяет изменения.
- **Что входит в пак:**
  - `install.md` — манифест: метаданные агента, список файлов для копирования, строки для вставки в `AGENTS.md` / `CLAUDE.md` / `director/core.md` / `routing-patterns.md`, сообщение пользователю после установки.
  - `core.md`, `CLAUDE.md`, `overrides.md`, `memory.md`, `failures.md` — стандартный набор агента.
  - `knowledge/`, `skills/`, `templates/` — если нужно.
- **Что делает установка автоматически:**
  - Копирует файлы агента в `office/agents/<name>/` (не затирает `overrides.md` / `memory.md` / `failures.md` если они уже есть).
  - Добавляет строку в `office/AGENTS.md`.
  - Добавляет `@office/agents/<name>/core.md` в layered include корневого `CLAUDE.md`.
  - Дописывает триггеры в `office/agents/director/core.md` и `director/knowledge/routing-patterns.md`.
  - Показывает пользователю живое сообщение о подключении нового помощника.
- **Базовый шаблон всегда минимальный** — Director + Strategist + Demiurg. Остальное приходит через паки по мере готовности пользователя.

Какие паки уже готовы — лежат в `_agent-packs/`. Сейчас там: **designer** (первый пример пака). Дальше будут: copywriter, sales, tech-lead.

## 1. Модульность

Каждый агент — самодостаточный модуль. Всё что нужно для интеграции:
- Создать папку `office/agents/<name>/` с файлами: `core.md`, `overrides.md`, `memory.md`, `failures.md`, `CLAUDE.md` (склейка `@core.md` + `@overrides.md`), плюс папки `knowledge/` и `skills/` если нужны.
- Добавить строку в `office/AGENTS.md`.
- Добавить `@office/agents/<name>/core.md` в layered include корневого `CLAUDE.md`.

Всё. Агент встроен.

## 2. Нет хардкода количества

Нигде не писать «Core Team из N», «в офисе четверо», «команда из трёх», «пока трое — я, Director, Стратег». Количество агентов меняется по мере установки — любая цифра в тексте сломается.

Правильно: «команда помощников», «актуальный состав — в `AGENTS.md`», «прочитай `AGENTS.md` и перечисли».

## 3. Свою роль — да, чужие зоны — нет

Агент пишет в своём `core.md` ТОЛЬКО про себя:
- Роль и зона ответственности
- Когда его зовут (триггеры)
- Что делает и что не делает
- Связки с другими агентами по именам ролей
- Guardrails, память, язык пользователю

Агент НЕ пишет «в офисе пока нет копирайтера» — эта информация живёт в `AGENTS.md`, не дублируй.

## 4. Связки через роли, не через количество

Handoff формулируется: «→ Copywriter (если есть в `AGENTS.md`)». Если агента нет — честно говорить пользователю: *"эту часть пока делаю я сам, специалиста подключим отдельным шагом"*.

Никогда: «→ наш копирайтер сделает» (если его нет в AGENTS.md) — это выдумка.

## 5. Layered design

Структура файлов одна для всех агентов:
- `core.md` — ядро. Обновляется централизованно при апдейтах шаблона.
- `overrides.md` — персональные правила пользователя. Приоритет над core. Не перезаписывается при апдейтах.
- `memory.md` — append-only память агента (decisions / patterns / context).
- `failures.md` — append-only журнал ошибок, grep перед каждой задачей.
- `CLAUDE.md` — склейка `@core.md` + `@overrides.md`. Claude Code читает это.

Пользователь правит только `overrides.md`. Обновления шаблона приходят через `/update-office` — затрагивают только `core.md`.

## 6. Триггеры в Director

При добавлении нового агента в офис:
- В `director/core.md` (секция Роутинг) добавить базовый триггер: «[ключевые слова] → [Агент]».
- В `director/knowledge/routing-patterns.md` добавить расширенный паттерн с примерами запросов и флоу handoff.
- Если нужна специальная секция (триггер-блок с алгоритмом handoff — например, дизайн-триггер) — описать её прямо в `install.md` пака, и при установке она добавится в `director/core.md`.

Это централизованная работа — строки триггеров описываются в `install.md` пака, скилл `install-agent` их вставляет. Агент сам себе триггеры в Director не прописывает.

## 7. Без самостоятельного подключения

Агент **НЕ** создаёт других агентов. Если пользователь просит нового — эскалация: проверить `_agent-packs/` на наличие пака, и если есть — предложить `/install-agent <name>`. Если пака нет — честно сказать что помощник пока не готов, перечислить текущий состав из `AGENTS.md`, предложить временное решение через существующих.

Demiurg тоже не создаёт агентов — он помогает со структурой офиса, но не конструирует новых членов команды. Установка готового пака — это работа скилла `install-agent`, не Demiurg'а.

## Чеклист при создании нового пака (для владельца шаблона)

При подготовке нового Agent Pack в `_agent-packs/<name>/`:

- [ ] Создана папка `_agent-packs/<name>/` со стандартным набором файлов (`core.md`, `CLAUDE.md`, `overrides.md`, `memory.md`, `failures.md`, `knowledge/`, `skills/`, `templates/` если нужно)
- [ ] Написан `install.md` с полными метаданными: agent_id, agent_name_human, список файлов, строки для `AGENTS.md`, корневого `CLAUDE.md`, `director/core.md`, `routing-patterns.md`
- [ ] В `install.md` — сообщение пользователю после установки живым языком (без IT-терминов)
- [ ] `core.md` написан в модульном стиле — про себя, без упоминаний количества команды
- [ ] Проверено что в новом `core.md` нет фраз «команда из N», «в офисе N», «пока N — я и ...»
- [ ] Связки с другими агентами сформулированы через роли с оговоркой «если есть в `AGENTS.md`»
- [ ] Пользователь-шаблон (`overrides.md`) пустой с подсказками, пользователь пишет туда сам
- [ ] Пак протестирован: `/install-agent <name>` на свежем клоне шаблона отработал без ручных правок
